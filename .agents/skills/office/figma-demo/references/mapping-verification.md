# 映射验证协议 — Mapping Verification Protocol

> 共享 reference。被 `/figma-demo` Phase 2 引用。
> 定义 Socratic 映射验证的完整规则、评分标准、歧义处理。

---

## 协议目的

映射验证不是走过场。它的目的是在调度 Builder 之前，确保 Orchestrator
对每个节点的理解达到了**能指导像素级还原的深度**。

表面理解 = "这是一个列表页"
深度理解 = "销售主管在出差途中打开这个列表，
他的注意力在搜索栏和最近联系的客户排序，
列表项的视觉重心是客户名+最后跟进时间，
hover时需要突出该行因为他可能在颠簸的车里用触摸板"

只有深度理解才能让 Builder 做出正确的细节决策。

---

## 5 问体系

### Q1：用户处境（User Situation）

**问：** 这个节点打开时，用户在做什么事情？他的注意力状态、
情绪状态、物理环境是什么？

**通过标准：**
- 描述了用户的**活动上下文**（不是功能描述）
- 描述了注意力状态（聚焦/分散/焦虑/平静）
- 描述了至少一个环境细节（时间/设备/场景）

**不通过的回答模式：**
- "用户查看列表" → 功能描述，不是处境
- "用户需要找到客户" → 任务目标，不是处境
- "用户打开了这个页面" → 循环定义

**通过的回答示例：**
- "销售代表刚结束客户拜访，在出租车上，需要在 3 分钟内录入拜访记录。
  注意力碎片化，手指操作精度低，情绪上可能因为谈判结果有波动"
- "销售主管周一早上 9 点开始一天工作，打开日报，
  心态是快速扫描而非深度阅读，需要在 5 分钟内掌握团队整体状况"

### Q2：视觉重心（Visual Focus）

**问：** 这个节点的视觉重心在哪里？用户视线第一个落点是什么？
这个落点和用户当前任务目标一致吗？

**通过标准：**
- 指明了具体的 UI 元素（不是"页面中间"）
- 解释了为什么这个元素是视觉重心
- 评估了重心与任务目标的一致性

**不通过的回答模式：**
- "视觉重心在页面中央" → 太模糊
- "主要内容区" → 没有指定具体元素
- "都很重要" → 没有层级判断

**通过的回答示例：**
- "视觉重心是搜索栏（顶部、全宽、L1字号占位符）
  ——因为用户进来就是找人的。搜索栏下方的最近联系列表是第二重心。
  两者都与'快速找到目标客户'的任务一致"

### Q3：过渡预期（Transition Expectation）

**问：** 从前一个节点到这个节点，用户的心理预期是什么？过渡动画是否匹配这个预期？

**首节点可跳过此问。**

**通过标准：**
- 描述了用户触发跳转时的心理预期（"深入"/"返回"/"打开工具"/"确认结果"）
- 评估了过渡动画类型与预期的匹配度
- 如果不匹配，说明了原因或建议调整

**不通过的回答模式：**
- "用 slide 动画" → 没有解释为什么
- "过渡效果合适" → 没有分析预期

**通过的回答示例：**
- "用户点击了客户名，
  心理预期是'进入这个客户的详细信息'——是一个'深入'的动作。
  slide-left（向左推进）传达'深入下一层'的空间隐喻，匹配。如果用 fade，
  会感觉像'切换到并列的内容'，不匹配'深入'的预期"

### Q4：状态完整性（State Completeness）

**问：** 这个节点有几种状态？每种状态的触发条件是什么？有没有遗漏的边界状态？

**通过标准：**
- 列出了所有明确的状态（来自口述+Figma）
- 列出了隐含的状态（来自交互补全规则）
- 对每种状态都说明了触发条件
- 对可能遗漏的状态做了判断（需要/不需要+原因）

**不通过的回答模式：**
- "有默认态和 hover 态" → 太少，没检查边界
- "状态都在 spec 里" → 没有独立思考

**通过的回答示例：**
- "4 种状态：默认态（有数据，页面加载完成）/ 空态（新客户无跟进记录，
  显示引导）/ 加载态（首次打开或下拉刷新，skeleton）/
  hover态（列表项鼠标悬停高亮）。
  权限不足态不需要——Demo不涉及权限场景。
  离线态不需要——汇报场景一定有网络"

### Q5：必要性（Necessity）

**问：** 如果删掉这个节点，整个流程还通不通？如果通，这个节点存在的理由是什么？

**通过标准：**
- 做了"删掉后会怎样"的思想实验
- 给出了明确结论（必要/非必要但有价值/可删除）
- 必要性的理由要和用户体验或汇报叙事相关

**不通过的回答模式：**
- "这个节点很重要" → 没有分析
- "不能删" → 没有原因

**通过的回答示例：**
- "删掉可以——用户从列表直接跳到详情也说得通。
  但保留这个过渡页的理由是：在汇报时，
  它展示了'从概览到细节'的导航逻辑，让观众理解信息架构的层级关系。
  如果跳过，观众可能会问'怎么突然就到了这个页面'"

## 评分规则

每个问题评分：PASS / WEAK / FAIL

```
PASS：回答符合通过标准，具体、有洞察
WEAK：回答部分符合，但缺少细节或有模糊处
FAIL：回答不符合标准，属于表面理解
```

### 节点级判定

```
5 PASS          → 节点通过
4 PASS + 1 WEAK → 节点通过，WEAK 项记入 mapping-proof.md 备查
3 PASS + 2 WEAK → 节点通过，但标记为"理解深度一般"
任何 FAIL       → 节点不通过，标记 NEEDS_CLARIFICATION
2+ WEAK（无FAIL）→ 节点通过，但需要在 spec.md 中补充额外说明
```

### Demo 级判定

```
所有节点通过                → 进入 Phase 3
有 NEEDS_CLARIFICATION 节点 → 汇总歧义，一次性向用户确认
超过 50% 节点有 WEAK        → 整体理解深度不足，考虑回到 Phase 1 重新翻译
```

## 歧义分类

### 可自行解决的歧义（不问用户）

```
- Figma 中按钮没有标注 hover 态 → 按补全规则添加
- 口述说"大概四五个" → 取 5 个
- 间距在 Figma 中不完全对齐 → 取最近的标准档
- 口述未提及空态 → 按补全规则添加
```

### 必须确认的歧义（问用户）

```
- 口述的节点数量 ≠ Figma 画面数量
- 口述提到了 Figma 中不存在的 UI 元素
- 两张 Figma 截图极度相似，无法判断对应关系
- 口述中有逻辑矛盾（A→B→A 循环但没说何时终止）
- Figma 中有多种设计方案（标注了 v1/v2），不确定用哪个
- 口述说"跟上次一样"但没有上次的上下文
```

---

## 歧义确认的交互规则

1. **一次性列出所有歧义** — 不逐个问，不多轮打扰
2. **每个歧义给出你的推断** — 不是问"是什么"，而是问"我理解的对吗"
3. **提供选项** — 如果有多种可能，列出选项让用户选
4. **歧义数量上限** — 如果 > 5 个歧义，说明 Phase 1 翻译质量不够，回退重做
