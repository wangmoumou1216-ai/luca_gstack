## 品味锚点检查

## 效率维度（4-5 锚点）

### 锚点 1 · Ryo Lu — 任务对齐（所有场景必检）

每个 UI 元素必须能回答「它在帮用户完成什么任务」。

```
逐一检查主要 UI 元素：
| 元素/区域 | 帮用户完成的任务 | 是否有「展示功能强大」的嫌疑 |
|---------|--------------|--------------------------|
| {元素} | {任务} | 无 / 有：{描述} |
```

**不通过动作：** 删除该元素。不重新设计，不缩小，不折叠——直接删除。

---

### 锚点 2 · Notion — 入口克制（所有场景必检）

首屏主要操作入口 ≤ 3 个，无重复入口。

```
首屏操作入口盘点：
1. {入口名称} — {作用}
...（超过3个标注，并给出合并建议）

重复入口：{无 / 有：{描述，必须删除}}
```

**不通过动作：** 重新分组合并；低频操作塞到 overflow menu。

---

### 锚点 3 · Linear — 路径效率（所有场景必检）

核心任务路径 ≤ 3 步，操作完成结果即时可见。

```
P0 核心任务：{任务描述}
路径：{步骤1} → {步骤2} → ... → 完成
总步骤：{N} 步（超 3 步必须返工）
每步等待时间：{< 500ms 同步 / ≥ 500ms 需要进度指示}
结果可见位置：{用户注意力所在位置 / 页面顶部 toast / 其他页面（不合格）}
```

**不通过动作：** 找可消除的步骤；合并两步；异步改乐观 UI。

---

### 锚点 4 · Attio — 密度有层次（所有场景必检）

信息密度高但有层次。B2B 销售用户需要密度，但密度不能是意外的。

```
信息层次：
  L1（每屏 1-2 个，最重要）：{描述}
  L2（每屏 3-5 个，次重要）：{描述}
  L3（背景信息）：{描述}
  层次可辨认：{是 / 否，3 秒内用户能看出什么最重要吗}

L1 的视觉权重：{字号 / 字重 / 颜色 / 位置，如何让它优先被看到}
L2 和 L3 的区分稳定性：{跨页面稳定 / 不稳定（不合格）}
```

**不通过动作：** 重新规划信息层级；不要靠加空白/加卡片/加分割线美化。

---

### 锚点 5 · Raycast — AI 入口位置（场景 B/C 必检，场景 A 写 N/A）

AI 入口必须在任务上下文内；AI 介入后路径只能变短或持平，不能变长。

```
AI 功能：{有/无 — 无则本锚点 N/A}
AI 入口位置：{任务上下文内（字段旁/选中文本菜单/工具条） / 独立侧栏 / 浮球 / 新标签页}
AI 介入后路径：{变短 / 持平 / 变长（不合格）}
对比不用 AI 的路径：{AI 路径 M 步 vs 传统 N 步，M ≤ N 合格}
```

**不通过动作：**
- 独立 AI 入口 → 嵌入到具体任务上下文
- AI 路径变长 → 移除 AI（此场景不适合 AI）

## 信任维度（2 锚点，场景 B/C 强制）

### 锚点 6 · Perplexity — 可追溯

AI 输出必须可追溯：用户能看到 AI 凭什么得出这个结论，并能点击验证。

```
AI 功能：{有/无 — 无则本锚点 N/A}

逐一检查每个 AI 输出点：
| AI 输出位置 | 附带可点击 sources? | 推理可展开? | 不确定性表达方式 |
|-----------|-----------------|-----------|--------------|
| {位置} | {是/否} | {是/否} | {百分比 / 高中低 / hedging / 多候选 / 无（不合格）} |

整体结论：
- 无来源的 AI 输出：{个数，必须 = 0}
- 无不确定性表达的 AI 输出：{个数，必须 = 0}
```

**不通过动作：**
- 无来源 → 要么加来源，要么移除这个 AI 输出（不能裸展示）
- 无不确定性 → 按 `ai-native-design-framework.md` 第 5 节选一种方式补齐

**阻断级锚点，一票否决。**

---

### 锚点 7 · Granola — 诚实表达

AI 不装作确定。在模型能力边界附近，产品必须诚实地表达"我不太确定"、
"这是初步的"、"可能有误"。

```
AI 功能：{有/无 — 无则本锚点 N/A}

逐一检查 AI 输出的语气：
| AI 输出位置 | 语气类型 | 过度自信点 |
|-----------|--------|--------|
| {位置} | {肯定句 / hedging / 主动告知局限} | {无 / 有：描述} |

冷启动处理：
- 新用户 / 数据不足时 AI 的行为：{主动告知"数据不足" / 硬给结果（不合格）}

超出能力边界的请求：
- AI 的处理方式：{坦率拒答 / 硬生成（不合格）}
```

**不通过动作：**
- 过度自信 → 加 hedging 语言（"可能"、"根据有限数据"、"建议 human review"）
- 冷启动硬给结果 → 加"数据不足"态，不强行给 AI 结果
- 超边界硬生成 → 加"拒答"态

**阻断级锚点，B2B 用户对虚假自信零容忍。**

---

## 代理维度（1 锚点，含 agent 动作 / 场景 D 强制）

### 锚点 8 · Cursor — 用户控制权

当 AI 代替用户做事时，用户必须始终能**看见、暂停、接管、撤销**。Agent 不是黑箱。

```
是否涉及 agent 动作：{是/否 — 否则本锚点 N/A}

四项强制通过（任一不通过视为阻断，agent 功能不能上线）：

1. 可见性：Agent 执行过程中用户能看到什么
   → {实时可见 / 完成后汇总 / 黑箱（不合格）}

2. 暂停性：用户能随时按"暂停"让 agent 停下
   → {可暂停且状态保留 / 可暂停但重来 / 不可暂停（不合格）}

3. 接管性：从暂停状态用户能手动接管或修改参数后让 agent 继续
   → {完整接管 / 部分接管 / 只能重来（不合格）}

4. 可逆性：已执行的动作能 undo / rollback
   → {单步 undo / 整体 rollback / 无撤销（不合格）}
   → 涉及对外通讯（邮件/消息）的动作是否默认草稿模式：{是 / 否（不合格）}
```

**不通过动作：** 任意一项不通过，返回 `/design-brief` Phase 1 Layer D
重新设计。不能带病上线。

**阻断级锚点。**

```

| 反模式 | 存在? | 位置 | 修复建议 |
|-------|------|----|-------|
| 浮球 AI | {有/无} | | |
| 新标签页 AI | {有/无} | | |
| 无来源结论 | {有/无} | | |
| 无不确定性 | {有/无} | | |
| 单候选强推 | {有/无} | | |
| 静默执行 | {有/无} | | |
| 不可撤销 | {有/无} | | |
| 模糊按钮（"AI 优化"/"智能处理"） | {有/无} | | |
```

**处理规则：** Slop ≥ 3 项 → 整体品味不合格，必须返工；Slop 1-2 项 →
修复建议写入品味报告，产出标"警告"。
