## 设计坐标系

**必须解决的问题：**
- P0-001：销售填跟进记录耗时 3-5 分钟 / 条（来自 PRD P0 用户故事）
- P0-002：销售经常漏掉"下一步计划"字段，导致后续跟进断层
- P0-003：客户上下文（上次沟通要点）不在视线中，销售要手动滚动查看

**可借鉴的范式：**
- 嵌入式预测（Gmail Smart Compose + Cursor Tab 补全）：在光标位置给 ghost text
- Inline Citation（Perplexity）：AI 输出旁的可点击角标 + hover 依据
- 多候选卡片（Attio AI 的"Why"展开依据）：下一步计划字段的多选

**不能碰的范围：**
- 不改详情页三栏布局
- 不增加新 Tab
- 不改跟进记录表单的字段顺序
- 不引入独立的"AI 助手"侧栏或浮球
- 不做 Agent 代替销售完整填写（需保留销售个性化叙事）
- 不涉及对外通讯（不生成发给客户的邮件/短信）

**AI Native 方向：**
- 结论：AI **介入**
- 范式：**嵌入式预测（Inline Prediction）**
- 介入节点：
  1. "跟进内容"字段：用户停顿 > 1.5s → 显示 ghost text 预测
  2. "下一步计划"字段：表单打开时预加载 3 个候选卡片
  3. "客户上下文摘要"：表单打开时在顶部显示 AI 生成的上次沟通要点

## 原生AI深度思考小结

**Layer A · 产品层：**
- 决策压缩：用户主动判断 **N = 2** 次（想客户上下文 + 定下一步）→ AI
  介入后 **N' = 0** 次纯主动判断
- 压缩：N - N' = **2**（刚好达到 ≥ 2 门槛）✅
- 采纳的 AI 介入：嵌入式预测（跟进内容 ghost text + 下一步计划多候选 + 上下文摘要）

**Layer B · 交互层：**
- 传统路径：**8 步**（见 native-design Phase 3 来源三）
- AI 原生路径：**4-5 步**
- 可行性：**高**
- 关键假设：LLM 响应时间 < 1 秒（高频场景对延迟敏感）
- Fallback：LLM 超时 / 不出 ghost text → 回退到传统手动填写路径

**Layer C · 信任层（AI 功能必填）：**
- 不确定性表达方式：**引用依据 + 多候选**
  - ghost text 右上角小角标 `[基于上次沟通]`，hover 显示依据，点击跳转历史记录
  - 下一步计划给 2-3 个候选，销售选（不是 AI 单独强推一个）
- 首次用户建立信任的路径：前 3 次使用时，ghost text 旁有 tooltip "按 Tab
  接受 / 继续打字拒绝"；3 次后不再提示
- AI 错了用户多久发现：**立即**（ghost text 出现的一瞬间销售就能判断）

**Layer D · 代理层：** **N/A** — 本功能不是场景 D（不涉及 agent 动作）

**对设计决策的影响：**
- 产品层决策压缩达标，继续进入交互层设计
- 交互层路径大幅缩短，但需要 LLM 响应 < 1 秒的技术前提 → 记录为假设挑战的关键假设
- 信任层的两种表达（引用依据 + 多候选）决定了 D-001 和 D-002 两条决策的具体形式

## 假设挑战结论

**核心假设：**
- **假设 1：** 销售愿意接受 ghost text 的"按 Tab 接受"交互范式（从未在 B2B CRM 场景被验证）
- **假设 2：** LLM 响应时间能稳定 < 1 秒（涉及 retrieve 客户历史 + embedding
  检索 + prompt 拼接 + 响应，当前技术栈不确定）

**最脆弱的假设：** 假设 1（假设 2 是技术问题，可优化；假设 1
是用户行为问题，如错就是根本性错误）

**建议验证节点：**
- 假设 1：在 HTML 原型完成后，找 3-5
  个真实销售做可用性测试（看他们首次接触能否理解"按 Tab 接受"）
- 假设 2：后端 POC 阶段压测响应时间

**（本场景非 D）信任假设挑战：** N/A

**结论：**
方案继续，但带两个风险：
- 如果假设 1 在用户测试中失败 →
  降级为"候选卡片模式"（跟进内容字段也改成显示 2-3 个候选卡片，
  让用户点选，不用 Tab）
- 如果假设 2 在压测中失败 → 降级为"异步预测模式"（ghost text
  出现时带"生成中..."状态，允许用户在等待时继续打字）

**Fallback 方案已记录到 D-001 的"接受的 tradeoff"字段。**

### D-001：跟进内容字段的 ghost text 预测

- **决策内容：** 在"跟进内容"富文本字段内，用户停顿输入 > 1.5 秒后，
  显示灰色 ghost text 预测下一句内容。销售按 Tab 接受，继续打字 / 按 Esc
  拒绝。

- **设计理由：**
  - 来自 PRD P0-001："销售填跟进记录耗时 3-5 分钟 / 条" → 需要大幅降低输入成本
  - 来自 inspired-design 的 Gmail Smart Compose 借鉴：高频输入场景最有效的范式
  - 来自 native-design Phase 3 来源三：Step 5（输入跟进内容）占总耗时 60%，AI
    部分代劳可压缩至 1/3
  - 来自 ai-native-design-framework 第 1 节：
    嵌入式预测范式的成立前提（用户注意力所在位置 + 可无缝忽略 +
    接受成本 ≤ 手动成本）全部满足

- **排除的备选方案：**
  - **备选 1：候选卡片模式**（显示 2-3 个完整内容候选让销售挑） →
    不选原因：破坏输入流，销售每次打字都要停下来看 3 个卡片比较，
    比自己写还累。只有在假设 1 失败时才降级到此方案。
  - **备选 2：结构化模板**（表单显示"今天和谁谈了什么 / 谈了多久 /
    结果如何"三个子字段） → 不选原因：
    过度结构化丢失销售的个性化叙事（native-design 否定方向 2 的核心理由）
    ，销售的文字是 CRM 的核心资产。
  - **备选 3：无 AI 辅助**（保持现状） → 不选原因：违反 P0-001 目标。

- **接受的 tradeoff：**
  - 依赖 LLM 响应时间 < 1 秒的技术前提。如果假设 2 失败，
    降级为"异步预测模式"（ghost text 出现时带"生成中..."状态）。
  - Tab 键冲突：表单里 Tab 通常切换字段，这里 Tab 用于接受 ghost text。
    需要在 onboarding 的前 3 次使用用 tooltip 教育。
  - 新销售学习成本：首次接触 ghost text 可能困惑（3-5 次使用后熟悉，
    但第一次接触需要明确引导）。
  - **放弃了**"让用户明确感到 AI 在帮忙"的显著性——ghost text 是安静的，
    不张扬；销售可能不会主动对 AI 道谢或反馈，需要通过
    D-005（反馈入口）间接收集。

- **AI Native 判定引用：** 全局 AI 方向 = 嵌入式预测。
  本决策在"跟进内容"字段旁显示 ghost text 预填建议，用户按 Tab
  接受——完整体现嵌入式预测范式，不打断主路径。

- **状态覆盖：**
  - 默认态：用户停顿 > 1.5s → ghost text 灰色（text-n11）叠加在光标后
  - 空态：新客户（历史 < 5 条）→ ghost text 功能关闭
  - 加载态：LLM 调用中 → ghost text 不显示（不打断用户）
  - 错误态：LLM 失败 → ghost text 静默关闭，不弹错误
  - 思考中态：停顿 1.5s 后的 200ms 渐入动画
  - 低置信态：置信度 < 阈值 → ghost text 不显示
  - 拒答态：数据极度不足 → ghost text 全程关闭 + 顶部提示"建议手动回忆"
  - 幻觉兜底态：已保存后每条记录旁的👍👎反馈入口

- **PRD 约束引用：** prd-constraints.md 第 3.2 条"优化方向边界：降低填写成本"；
  第 4.1 条"Not-Do：不改表单字段顺序" — 本决策只增加 ghost text，
  不改字段顺序，符合约束。

### D-002：下一步计划字段的 3 个候选卡片

- **决策内容：** 表单打开时，"下一步计划"字段上方显示 3
  个候选卡片（横向排列），每个卡片含：动作类型（如"再约电话"）+
  时间建议（如"3 天内"）+ 依据（底部小字）。
  销售点击卡片后内容填入字段；点击字段后可自由编辑覆盖候选内容。

- **设计理由：**
  - 来自 PRD P0-002："销售经常漏掉下一步计划字段"
  - 来自 inspired-design 的 Attio AI 借鉴：多候选 + 可展开依据的 UI 模式
  - 来自 native-design Phase 3 来源三：Step 6（判断下一步）是判断型高负担，AI
    给候选能大幅降低认知负担
  - 来自 ai-native-design-framework 第 5 节："多候选并列"是不确定性表达的合法方式之一

- **排除的备选方案：**
  - **备选 1：AI 直接填入（单候选）** → 不选原因：违反 native-design
    否定方向 2 的底线——销售要保留对下一步的判断权，
    单候选会让销售习惯性接受而不思考。
  - **备选 2：对话式询问**（"下一步你想做什么？" →
    销售选"再约/推进/归档"） → 不选原因：多一次对话交互，路径变长；
    违反 Raycast 锚点。
  - **备选 3：保持现状（纯手动填写）** → 不选原因：违反 P0-002 目标。

- **接受的 tradeoff：**
  - 3 个候选占表单空间（约 120px 高度）→ 表单整体高度增加约 15%，
    但在表单弹窗的合理范围内。
  - AI 生成候选需要 LLM 调用 → 加载时需要 skeleton 态。
  - **放弃了**"AI 一键智能填写"的极致简化——但这正是我们要放弃的，
    因为保留销售的选择权是信任层的核心。

- **AI Native 判定引用：** 全局 AI 方向 = 嵌入式预测。
  本决策在"下一步计划"字段提供多候选预测，
  用户单击选择——体现嵌入式预测的多候选变体，
  同时符合信任层的"多候选并列"不确定性表达。

- **状态覆盖：**
  - 默认态：3 个候选卡片横向排列
  - 空态（新客户）：显示行业通用模板 3 张 + 小字"当前候选基于行业数据"
  - 加载态：3 个卡片占位骨架（animate-pulse）
  - 错误态：候选卡片区域隐藏，用户直接手动填写
  - 低置信态：只显示 1-2 个候选 + "暂无更多建议"说明
  - 拒答态：数据极度不足 → 只显示行业通用模板，小字"建议手动填写"

- **PRD 约束引用：** prd-constraints.md 第 3.2 条"优化方向边界：减少漏填"；第
  4.2 条"Not-Do：不改字段结构" — 本决策只在字段上方加候选卡片，
  字段本身结构不变。

### D-003：表单顶部的客户上下文摘要

- **决策内容：** 跟进表单弹窗打开时，顶部显示 AI
  生成的"上次沟通要点"（3-5 行），每个要点结尾带 `[1]` 角标，hover
  显示完整原文摘要，点击跳转到对应历史跟进记录。
  摘要区右上角有"查看完整历史"链接。

- **设计理由：**
  - 来自 PRD P0-003："客户上下文不在视线中，销售要手动滚动查看"
  - 来自 inspired-design 的 Perplexity 借鉴：inline citation 的信任层标杆做法
  - 来自 native-design Phase 3 来源二：用户心智是"操作型"，
    但对"上下文可见"有强需求——AI 把上下文前置能减少记忆负担
  - 来自 ai-native-taste-anchors 锚点 6 (Perplexity)：AI 输出必须可追溯，citation
    模式是成本最低的实现

- **排除的备选方案：**
  - **备选 1：纯文字摘要（无 citation）** → 不选原因：违反 Perplexity 锚点，
    用户无法验证 AI 摘要是否真的来自历史记录。
  - **备选 2：AI 摘要弹窗（用户点"查看摘要"才出现）** → 不选原因：
    用户不会主动点，摘要价值归零；且增加 1 步操作。
  - **备选 3：显示完整历史记录**（不做 AI 摘要，
    只把历史时间线放到表单顶部） → 不选原因：
    占用太多空间（表单高度不够），且未真正解决"认知负担高"的问题。

- **接受的 tradeoff：**
  - 摘要质量依赖 LLM 质量：可能生成不准确的摘要。通过 citation
    可追溯机制缓解（用户点击能验证）。
  - 占用表单顶部约 80px 空间：表单整体往下推。
  - 新客户场景下显示"暂无 AI
    摘要"可能被认为是"产品缺陷"——需要拒答态文案温和，避免负面感受。

- **AI Native 判定引用：** 全局 AI 方向 = 嵌入式预测。
  本决策虽然不是典型的"预测用户下一步输入"，
  但属于嵌入式预测的前置形态——"预测用户进入表单后最需要看的信息"，
  并将其前置到视线中心。

- **状态覆盖：**
  - 默认态：3-5 行摘要 + citation 角标
  - 空态（新客户）：显示"这是首次跟进，暂无历史上下文"
  - 加载态：3 行 skeleton（animate-pulse）
  - 错误态：显示"暂无 AI 摘要"（不阻断表单填写）
  - 拒答态：历史 < 2 条时显示"数据不足，建议手动回忆上下文"

- **PRD 约束引用：** prd-constraints.md 第 3.2 条"优化方向边界：降低认知负担"。

### D-004：ghost text 的 onboarding tooltip（前 3 次使用）

- **决策内容：** 新用户首次使用本功能时，ghost text 首次出现时附带
  tooltip："按 Tab 接受 AI 建议 / 继续打字拒绝"。前 3
  次使用后不再显示（localStorage 记录）。

- **设计理由：**
  - 来自 native-design Phase 3 来源二：ghost text 的触发机制对新用户不可预测，需要明确引导
  - 来自 inspired-design 的 Cursor 借鉴：Cursor 用户靠 Tab
    接受补全（已有行业共识），但销售场景需要额外教学
  - 来自 ai-native-design-framework 第 4 节 Layer C：用户凭什么相信
    AI——需要"首次使用能建立信任的路径"

- **排除的备选方案：**
  - **备选 1：无 onboarding**（直接出现 ghost text） → 不选原因：违反
    native-design 来源二子问题 3 的"最容易引发困惑的节点"预防。
  - **备选 2：弹窗教学**（首次打开功能时弹出完整教程） → 不选原因：
    销售讨厌被弹窗打断，会立即关闭而不看。
  - **备选 3：永久显示 tooltip** → 不选原因：熟悉用户被反复打扰，违反 Ryo Lu 锚点。

- **接受的 tradeoff：**
  - 依赖 localStorage 记录使用次数：清缓存 / 换设备后会重置。可接受（销售极少换设备）。
  - 可能前 3 次销售都点了 tooltip 之外的区域，使 tooltip 消失。通过"累计 3
    次成功使用 Tab"计数，不是"3 次打开表单"计数。

- **AI Native 判定引用：** 本决策服务于 D-001（ghost text 预测）
  的首次信任建立，不独立涉及 AI 介入节点。

- **状态覆盖：** N/A — 仅前 3 次显示的一次性 UI，后续状态与 D-001 合并

- **PRD 约束引用：** N/A — 属于 D-001 的配套体验优化，不独立对应约束条目

### D-005：已保存跟进记录的 AI 反馈入口

- **决策内容：** 详情页的跟进记录时间线上，每条记录（如果有 AI
  辅助生成的部分）右侧显示小号 👍/👎 图标（默认灰色）。点👎
  展开原因选择："无依据 / 过时 / 理解错意图 / 其他"，
  记录用户反馈用于后续模型优化。

- **设计理由：**
  - 来自 ai-native-state-coverage 第 6 状态（幻觉兜底态）：AI 输出需要反馈机制
  - 来自 ai-native-taste-anchors 锚点 6 (Perplexity) + 锚点 7 (Granola) 的信任层要求
  - 来自 PRD P0 中"长期希望 AI 质量持续改善"的隐含需求

- **排除的备选方案：**
  - **备选 1：无反馈入口** → 不选原因：违反 ai-native-state-coverage 第 6 状态的强制要求。
  - **备选 2：弹窗询问**（每次保存后弹"这次 AI 帮助怎么样？"） →
    不选原因：打扰高频操作流，销售很快会关闭提示弹窗。
  - **备选 3：仅 thumbs up**（只收集正向反馈） → 不选原因：
    负向反馈才是产品改进的关键。

- **接受的 tradeoff：**
  - 入口默认灰色不显眼 → 销售可能完全不用。可接受，因为强制反馈会更差。
  - 反馈数据的价值依赖于用户使用意愿 → 早期数据可能稀疏。

- **AI Native 判定引用：** 全局 AI 方向 = 嵌入式预测。
  本决策不直接参与预测生成，但是配套的信任层反馈回路。

- **状态覆盖：**
  - 默认态：灰色 👍/👎 图标
  - 反馈后：文字变为"已记录"（text-n11 text-12），1.5s 后消失
  - 点👎 展开的原因选择：dropdown 样式

- **PRD 约束引用：** N/A — 属于长期质量保证机制，不直接对应单个约束条目
