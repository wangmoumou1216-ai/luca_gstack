## 设计坐标系

**必须解决的问题：**
- P0-001：销售每周花 2-3 小时手动整理新线索（分类、打标签、分配优先级）
- P0-002：手动整理导致高优先级线索响应延迟（平均延迟 18 小时 vs 行业标杆 4 小时）
- P0-003：分类标准不一致——同一种线索，不同销售分类到不同类别

**可借鉴的范式：**
- Cursor Agent 的 plan preview + step approval（先看计划再执行）
- Linear 的 AI batch operations（批量操作 + review before apply）
- Claude Code 的 tool use 审批流（每个危险动作需要用户批准）

**不能碰的范围：**
- 不改动线索的数据模型（字段结构不变）
- 不自动发送任何通知/邮件给线索客户（**对外通讯强制草稿模式**）
- 不自动删除/归档任何线索（Agent 只分类和标注，不做破坏性操作）
- 不替代销售做最终的跟进决策
- 不在没有销售确认的情况下把线索分配给其他人

**AI Native 方向：**
- 结论：AI **介入**
- 范式：**代理式执行（Agentic Execution）**
- 介入节点：
  1. 每周一早 9 点自动触发"本周新线索整理"任务
  2. Agent 执行：查询未分类线索 → 按规则 + AI 判断分类 → 打标签 → 排优先级
  3. Agent 产出：一份"整理计划"草稿 → 销售审批后写入
- **Agent 化边界声明（场景 D 强制）：**
  - Agent 可自主做：查询数据、生成分类建议、排列优先级
  - Agent 必须每次确认：修改线索分类字段、修改优先级、标记为"已整理"
  - Agent 绝不可做：删除线索、发通知给客户、把线索分配给其他销售

## 原生AI深度思考小结

**Layer A · 产品层：**
- 决策压缩：用户主动判断 **N = 47**（每条线索都要手动判断分类 + 优先级，
  平均每周 47 条新线索）
- AI 介入后 **N' = 3-5**（销售只审批"整理计划"里的异常项 + 确认最终写入）
- 压缩：N - N' = **42-44**（远超 ≥ 2 门槛）✅

**Layer B · 交互层：**
- 传统路径：打开线索列表 → 逐条看 → 逐条改分类 → 逐条改优先级 →
  完成。每条 3-5 分钟 × 47 条 = 约 3 小时
- AI 原生路径：Agent 自动执行 → 销售收到"整理计划" → 审批 3-5 分钟 →
  确认写入。**共 3 步（收到 → 审批 → 确认），约 5-10 分钟**
- 路径对比：**3 步 << 47×5 步**（路径大幅缩短）✅
- 关键假设：Agent 的分类准确率 ≥ 85%（低于这个值，销售审批时间会暴增）
- Fallback：Agent 不可用时 → 回到手动逐条分类

**Layer C · 信任层（AI 功能必填）：**
- 不确定性表达方式：**预览-执行分离 + 置信度**
  - Agent 产出"整理计划"后，销售看到每条线索的"AI 分类 + 置信度（高/中/低）+ 依据"
  - 低置信度线索高亮标注，提醒销售重点审核
  - 销售可以逐条修改分类后再确认
- 首次用户建立信任的路径：第 1 周手动整理 + Agent
  对照结果（让销售看到"如果用 Agent 会怎样"）
- AI 错了用户多久发现：**审批时立即发现**（错分类在审批计划里就能看到）

**Layer D · 代理层（场景 D 强制，四项全填）：**

```
1. 过程可见性：
   → Agent 执行过程中，线索列表页顶部显示"AI 正在整理线索（已处理 {N}/{M}）"进度条
   → 点击进度条展开：已完成步骤 + 当前步骤 + 待执行步骤
   → 完成后：进度条变为"整理完成，请审批计划"按钮
   可见性评级：实时可见 ✅

2. 暂停性：
   → 进度条右侧有"暂停"按钮
   → 暂停后：已整理的线索保留，未整理的暂停
   → 暂停状态显示："已暂停在第 {N} 条，剩余 {M} 条未处理"
   → 恢复按钮："继续整理" / "从暂停处重新开始"
   暂停性评级：可暂停且状态保留 ✅

3. 接管性：
   → 暂停后，销售可以选择"手动完成剩余线索"
   → Agent 已分类的保留，销售手动处理 Agent 未触及的
   → 或销售可以修改 Agent 已分类的结果后让 Agent 继续
   接管性评级：完整接管 ✅

4. 可逆性：
   → "整理计划"未确认前：随时可取消，Agent 所有操作零写入
   → "整理计划"确认写入后：7 天内可一键"撤销本次整理"
   → 撤销粒度：整体回滚（恢复到整理前状态）+ 单条回滚（只撤销某几条）
   → 对外通讯：Agent 不发送任何通知给客户（绝不可做），无需草稿模式
   可逆性评级：整体 + 单条 undo ✅
```

**对设计决策的影响：**
- Layer D 四项全通过，Agent 化改造可行
- 信任层的核心是"预览-执行分离"——整理计划必须审批后才写入
- 技术前提：Agent 需要后台运行 + 持久化执行状态 + 断点续传能力

## 假设挑战结论

**核心假设：**
- **假设 1：** Agent 的分类准确率 ≥ 85%（低于此值，审批成本会超过手动分类）
- **假设 2：** 销售愿意从"逐条手动"切到"Agent 整理 + 我审批"的工作模式

**最脆弱的假设：** 假设 2（假设 1 是技术问题；假设 2 是行为问题）

**【场景 D 专有】信任假设挑战：**
- 假设："销售会信任 Agent 自动分类他们的线索"
- 若用户不信任：
  - Fallback A：降级为"决策增强"范式 → Agent 不自动分类，
    只给每条线索"建议分类 + 依据"，销售逐条点"采纳"
  - Fallback B：增加"第 1 周对照期" → Agent 在后台跑但不写入，
    只给报告"如果用 Agent 会怎样分类"，和销售手动结果对照

**建议验证节点：**
- 假设 1：用历史 3 个月数据做回测，算准确率
- 假设 2：找 3 个不同风格的销售做 1 周试点（保守型 / 激进型 / 中立型）

**结论：** 方案继续，但内置 Fallback B（第 1 周对照期）作为默认首次体验。

## 品味检查四锚点

### 效率维度

- **Ryo Lu**：每个 UI 元素能回答「帮用户完成什么任务」？→ **是**
  - 进度条 → 帮用户实时知道 Agent 进度
  - 暂停按钮 → 帮用户在不确定时紧急停车
  - 整理计划审批页 → 帮用户一次性审核所有 Agent 结果
  - 撤销按钮 → 帮用户在事后发现问题时回滚
  - **没有"看着很酷但不知道干嘛"的元素**

- **Linear**：核心任务路径步骤数？→ **3 步**（收到计划 → 审批 → 确认）
  ；能再少？→ **否**（审批步骤不能删，否则变成黑箱 Agent）

- **Attio**：信息层次 L1/L2/L3 可辨认？→ **是**
  - L1：每条线索的公司名 + AI 推荐分类（text-15 font-medium）
  - L2：分类依据 + 置信度（text-13 text-n19）
  - L3：线索来源 / 创建时间 / 历史交互次数（text-13 text-n11）

- **Notion**：首屏主要操作入口数量？→ **2 个**（确认写入 + 取消整理）；
  有无重复？→ **无**

- **Raycast**：AI 入口在任务上下文内？→
  **是**（"本周线索整理"按钮在线索列表页顶部，不是独立页面）；AI
  介入后路径变长？→ **否**（3 步 << 原 47×5 步）

### 信任维度

- **Perplexity**：AI 输出可追溯？→ **是**
  - 每条线索的"AI 分类"旁有"查看依据"链接
  - 展开显示：依据 1（行业匹配度 80%）+ 依据 2（近 30 天 3 次网站访问）+

- **Granola**：AI 诚实表达不确定性？→ **是**

  - 无法分类的线索标记"需人工判断"（拒答态），不强行归类
  - 第 1 周默认"对照期"（Agent 不写入，只对照）
  - 过度自信点：**无**

### 代理维度（场景 D 强制）

## 设计决策清单

### D-001：Agent 执行过程的实时进度条

- **决策内容：** 线索列表页顶部在 Agent 运行时显示进度条（"AI
  正在整理线索 {N}/{M}"），可展开查看每步状态（✅/⏳/⏸️），
  右侧有"暂停"按钮。

- **设计理由：**
  - 来自 ai-native-design-framework 第 4 节 Layer D：Agent 动作过程必须可见
  - 来自 ai-native-taste-anchors 锚点 8 (Cursor)：可见性是四项控制权的第一项
  - 来自 PRD P0-001：Agent 替代了原来 2-3 小时的手动工作，用户需要知道进展

- **排除的备选方案：**
  - **备选 1：后台静默执行 + 完成后推送通知** → 不选原因：违反 Cursor
    锚点的可见性要求。用户在执行期间不知道 Agent 在做什么，
    如果出错了更不知道错在哪。
  - **备选 2：全屏 Agent dashboard** → 不选原因：Agent
    执行时销售还要做其他事情（打电话、开会），不能强占整屏。进度条 +
    可展开是最小干扰方式。

- **接受的 tradeoff：**
  - 进度条占据列表页顶部 40px 空间 → 在 Agent 运行的 3-10 分钟内影响信息密度，可接受
  - 实时更新进度需要 WebSocket / SSE → 技术复杂度增加

- **AI Native 判定引用：** 全局 AI 方向 = 代理式执行。本决策直接服务于 Layer
  D 的可见性要求。

- **状态覆盖：**
  - 默认态：无进度条（Agent 未运行）
  - 思考中态：进度条显示"AI 正在整理..."
  - 部分完成态：进度条显示 ✅/❌/⏸️ 混合状态
  - Agent 执行中态：进度条蓝色脉冲 + 当前步骤文字

- **PRD 约束引用：** prd-constraints.md 第 5.1 条"Agent 执行过程必须可见"

### D-002：Agent 的暂停 + 接管机制

- **决策内容：** 进度条右侧"暂停"按钮。暂停后显示三个选项："恢复" /
  "手动完成剩余" / "取消整理"。选"手动完成"时，Agent 已整理的保留，
  剩余线索变为手动编辑模式。

- **设计理由：**
  - 来自 ai-native-design-framework 第 4 节 Layer D：暂停性 + 接管性
  - 来自 ai-native-taste-anchors 锚点 8 (Cursor)：暂停后状态保留 + 完整接管
  - 来自假设挑战：假设 2（销售可能不信任 Agent）的 Fallback A 前提是用户能随时接管

- **排除的备选方案：**
  - **备选 1：只能取消，不能暂停恢复** → 不选原因：Agent 可能已经整理了
    30/47 条，全部取消太浪费。暂停 +
    恢复让用户可以"看一眼对不对"然后继续。
  - **备选 2：暂停后必须重来** → 不选原因：违反接管性要求。
    用户暂停是因为"想看看到目前为止对不对"，不是"想从头来"。

- **接受的 tradeoff：**
  - 暂停后的状态持久化增加后端复杂度（需要断点续传机制）
  - "手动完成剩余"模式下的 UI 混合态（Agent 整理的 + 手动的 混在同一列表）需要视觉区分

- **AI Native 判定引用：** 全局 AI 方向 = 代理式执行。本决策直接服务于 Layer
  D 的暂停性和接管性。

- **状态覆盖：**
  - 默认态：暂停按钮可见（Agent 运行时）
  - 暂停后：三选项浮层（恢复 / 手动完成 / 取消）
  - 接管后：列表页标记哪些是 Agent 已处理（绿色标签）、哪些待手动处理（灰色标签）

- **PRD 约束引用：** prd-constraints.md 第 5.2 条"Agent 必须可暂停、可接管"

### D-003：整理计划审批页（预览-执行分离的核心 UI）

- **决策内容：** Agent 完成后生成"整理计划"——一个全屏审批页面，
  表格形式显示每条线索的"原始状态 → AI 建议分类 + 置信度 + 依据"。
  低置信度行黄色高亮。销售可以逐行修改分类，底部"确认写入"按钮。

- **设计理由：**
  - 来自 ai-native-design-framework 第 5 节：预览-执行分离是 agent 场景的标准不确定性表达
  - 来自 ai-native-taste-anchors 锚点 6 (Perplexity)：每条 AI 分类必须有可追溯依据
  - 来自 ai-native-taste-anchors 锚点 7 (Granola)：低置信度必须诚实标注

- **排除的备选方案：**
  - **备选 1：列表页 inline 审批（不跳全屏）** → 不选原因：47
    条线索的审批信息量大，在原列表页上加审批 UI 会导致信息层级混乱。
    专用审批页保持视觉清晰。
  - **备选 2：AI 自动写入 + 事后通知** → 不选原因：
    违反"预览-执行分离"原则和 Cursor 锚点。场景 D 的核心设计就是"Agent
    做草稿，人做审批"。
  - **备选 3：邮件/消息发送审批计划** → 不选原因：审批在 CRM 内完成比跳到邮件更高效。

- **接受的 tradeoff：**
  - 全屏审批页打断了线索列表的浏览流 → 但审批本身就是一个独立任务，
    不应在列表页"顺便"做
  - 47 条线索的审批表格可能很长 → 加分页 / 分组（高置信 / 低置信 /
    待人工），销售可以先确认高置信部分

- **AI Native 判定引用：** 全局 AI 方向 = 代理式执行。
  本决策是"预览-执行分离"的核心载体——没有这个审批页，Agent
  就变成了黑箱执行。

- **状态覆盖：**
  - 默认态：审批表格（全量线索 + AI 分类 + 置信度 + 依据）
  - 低置信态：该行黄色背景 + "置信度：低" 标签
  - 待 Steer 态：该行显示 2 个候选分类卡片，销售必须选
  - 拒答态：该行显示"需人工判断" + 手动分类下拉
  - 幻觉兜底态：该行有"标记错误"按钮 + 原因选择
  - 成功态：确认后返回列表页 + 成功条

- **PRD 约束引用：** prd-constraints.md 第 5.3 条"所有 Agent 操作必须经审批后写入"

---

### D-004：确认写入后的撤销机制

- **决策内容：** 审批确认后，线索列表页顶部显示"本次整理 {N}
  条线索已写入"成功条 + "撤销"链接（7 天内有效）。
  点"撤销"后可选"整体回滚"或"选择性回滚"（勾选要撤销的行）。

- **设计理由：**
  - 来自 ai-native-design-framework 第 4 节 Layer D：所有 Agent 动作必须可 undo / rollback
  - 来自 ai-native-taste-anchors 锚点 8 (Cursor)：可逆性是四项控制权的最后一项

- **排除的备选方案：**
  - **备选 1：不可撤销（确认即最终）** → 不选原因：直接违反 Cursor
    锚点。Agent 错了销售要手动恢复 47 条，成本极高。
  - **备选 2：只支持整体回滚** → 不选原因：Agent 整理了 47 条，可能只有 3
    条错了。整体回滚要重新整理所有线索，太浪费。选择性回滚更精准。

- **接受的 tradeoff：**
  - 7 天撤销窗口占用后端存储（需保留操作前快照）
  - 选择性回滚的 UI 复杂度高于整体回滚

- **AI Native 判定引用：** 全局 AI 方向 = 代理式执行。本决策保障用户在 Agent
  出错时的最后防线。

- **状态覆盖：**
  - 默认态：成功条 + 撤销链接
  - 撤销中：确认弹窗"确定要撤销？" + 整体/选择性选项
  - 撤销完成："已撤销 {N} 条线索的分类变更"

- **PRD 约束引用：** prd-constraints.md 第 5.4 条"Agent 所有写入操作必须可撤销，保留 7 天"

### D-005：第 1 周对照期（信任建立机制）

- **决策内容：** 新用户首次启用"AI 整理线索"时，默认进入"对照期"：Agent
  在后台运行但不写入，只生成"如果用 Agent 会怎样分类"的对照报告。
  销售对比自己的手动结果和 Agent 建议后，选择"信任 Agent，
  开始自动整理"或"继续手动"。

- **设计理由：**
  - 来自假设挑战的 Fallback B：解决"销售不信任 Agent"的行为假设
  - 来自 ai-native-taste-anchors 锚点 7 (Granola)：首次用户没有信任基础，
    不能一上来就让 Agent 动数据
  - 来自 native-design Phase 3 来源二：销售是操作型心智，切到委托型需要渐进建立信任

- **排除的备选方案：**
  - **备选 1：直接启用 Agent（无对照期）** → 不选原因：
    假设挑战识别出"销售不信任 Agent"是最脆弱假设。无对照期 =
    强制用户信任 = 最可能的失败点。
  - **备选 2：永久对照模式（Agent 永远不自动写入）** → 不选原因：
    这就变回了"决策增强"而非"代理式执行"。对照期是渐进过渡，
    不是永久状态。

- **接受的 tradeoff：**
  - 第 1 周用户得不到 Agent 的全部价值（还是要手动整理）
  - 对照报告的 UI 开发成本

- **AI Native 判定引用：** 全局 AI 方向 = 代理式执行。
  本决策是"从操作型心智过渡到委托型心智"的桥梁设计，不是回退。

- **状态覆盖：**
  - 对照期默认态：列表页顶部"AI 对照期 (第 {N}/7 天)" + "查看本周对照报告"按钮
  - 对照期成功态：报告生成完成 + 对比表格
  - 过渡态："对照期结束，开启自动整理？" + "是，开始" / "再观察一周"

- **PRD 约束引用：** N/A — 属于信任层的体验设计，超越单条 PRD 约束

## 交接块（下游 skill 必读）

**本步决定了什么：**
- 线索整理 Agent 的完整交互流程：触发 → 执行（可见/可暂停）→
  审批计划 → 确认写入 → 可撤销
- AI Native 范式：代理式执行
- Layer D 四项控制权全部通过：可见（进度条）/ 可暂停（暂停按钮）/
  可接管（手动完成剩余）/ 可撤销（7天 undo + 单条/整体）
- 12 状态全覆盖（场景 D 不允许 N/A）
- Agent 授权边界：可自主查/分类 → 必须确认写入 → 绝不可删/发通知/分配

**下游 /html-prototype 需要知道：**
- 设计范围（来自 PRD）：线索列表页 + 整理计划审批页（两个页面）
- 框架类型：**列表页 — framework/list-page.html**
- 状态覆盖策略：**12 状态全部需要单独设计**（场景 D 硬约束）
- 映射表路径：见本文件「shadcn 组件映射表」节
- **AI 专有状态的 UI 形式** — 本决策场景 D，12 状态全部非 N/A。html-prototype
  必须额外生成全部 7 个 AI 专有状态页。

**下游 /html-prototype 不应该做：**
- 不应省略任何状态的实现 — **场景 D 不允许 N/A**
- 不应把 Agent 执行做成"黑箱后台任务"——必须有实时进度条
- 不应把审批设计成"一键全部确认"——必须逐行可修改
- 不应省略撤销机制
- 不应让 Agent 的对外通讯相关动作出现在 UI 里（已在 REMOVED 中记录）

**对 /html-prototype 的命令式指示（逐条执行）：**
1. 读取本文件的"体验验证结论"节 12 状态覆盖表
2. **12 个状态全部生成对应 UI**（用 display:none 切换 + `<!-- STATE: xxx -->` 注释）
3. 审批表格的低置信度行必须用 bg-warning-bg 高亮
4. 全局导航栏的 Agent 运行指示器必须在所有页面可见
5. 对照期 UI（D-005）作为首次使用的默认状态实现
