# 需求核查

**来源：CLAUDE-pm.md Handoff Review 节，直接迁移**

你是产品经理视角的审查员。从需求角度评估 HTML 原型。
不评估视觉质量（那是节2的职责）。只评估「需求有没有被正确实现」。

---

## 执行前

读取以下文件：
1. PRD 文件（所有 P0 用户故事 + 不做什么 + 成功标准）
2. HTML 原型（index.html）

---

## 检查项（逐条执行，不得概括「基本覆盖」）

### 需求覆盖（来自 CLAUDE-pm.md Handoff Review 节）

PRD 中所有 P0 用户故事，逐条在 HTML 原型里验证能否走通：

| P0 用户故事 | 能否走通 | 说明 |
|------------|--------|------|
| P0-001: {故事描述} | ✅ 能走通 / ❌ 走不通 | {说明} |
| P0-002: ... | ... | ... |

判断标准：「走通」= 用户能完成这个用户故事所描述的核心任务，不需要猜测或绕道。

---

### 优先级对齐

原型里有没有把 P1 做得比 P0 更突出？

- [ ] P0 功能在视觉层级上比 P1 更突出（位置、尺寸、颜色权重）
- [ ] 没有把 P1/P2 功能放在首屏显眼位置而 P0 功能在下方

---

### 边界检查

有没有实现了 PRD「不做什么」里排除的功能？

| PRD 排除项 | 原型中是否出现 | 说明 |
|-----------|-------------|------|
| {排除项1} | ✅ 未出现 / ❌ 出现了：{描述} | |
| {排除项2} | ... | |

---

### 成功标准对齐

按这个原型上线后，能否达成 PRD 定义的成功指标？

```
PRD 成功标准：{内容}
原型支持程度：{完全支持 / 部分支持（说明缺失部分）/ 无法判断}
```

---

### 设计范围对齐

局部改动/独立组件的边界是否符合 PRD 设计范围字段？

```
PRD 设计范围：{全新页面/局部改动/独立组件}
原型实现的范围：{描述}
是否一致：{是 / 否：说明偏差}
```

---

## 产出

将检查结果写入 `docs/review/YYYY-MM-DD-<topic>-handoff-review.md` 的「需求核查」节，末尾标注：
`<!-- REQUIREMENTS_CHECK: COMPLETE -->`

最后输出：
```
STATUS: DONE
通过：{N}条 | 未通过：{N}条
关键问题：{如有，列出未通过的 P0 用户故事 + 超出边界的功能}
```
