## 文档定位

这是一份**结构层（Structure Plane）**交互架构文档。

它回答的问题是：
- 用户从哪里来，到哪里去？
- 用户和系统之间的对话逻辑是什么？
- 控制权在谁手里，什么时候转移？
- 信任怎么从零建起？
- 最坏情况下系统怎么降级？

它**不做**的事：
- 不定义组件尺寸、颜色、字体（骨架层 → design-brief）
- 不定义视觉风格、品牌表达（表面层 → html-prototype）
- 不定义数据库结构、API接口（实现层 → 工程文档）

**评判标准：一个不了解项目上下文的设计师，读完这份文档后，
能画出完整的用户旅程图和交互流程图。如果不能——文档不够完整。**

## 2. 核心交互模型

| 方式 | 描述 | 适用场景 | 优先级 |
|------|------|---------|-------|
| {如"意图栏输入"} | {自然语言输入} | {所有场景} | {主要} |

| {如"GUI操作触发"} | {用户在传统GUI中的操作自然触发Agent} | {上下文相关} | {隐式} |
| {如"Agent主动推送"} | {Agent检测到条件后推送} | {异常/提醒} | {被动} |

### 2.2 系统响应模式

| 模式 | 触发条件 | 用户感知 |
|------|---------|---------|
| 即时场景切换 | 意图被高置信度理解 | 屏幕切换到新场景 |
| 需要澄清 | 意图模糊或有多种解读 | 系统追问一个澄清问题 |
| 拒绝并解释 | 意图超出Agent能力或权限 | 系统说明为什么不能做 + 替代建议 |
| 后台执行 | 长时间任务 | 系统确认已开始 + 进度可查 |

### 2.3 流程模板集

**每种任务类型一个模板。模板定义的是交互模式，不是具体页面布局。**

#### 模板 A：只读查询

```
触发 → 意图被理解
  → 系统组装查询结果场景
  → 呈现结果（用户扫描，无需操作）
  → 用户可以：
     ├─ 发新意图（切到其他场景）
     ├─ 基于结果发起操作（升级到模板B/C）
     └─ 关闭/回退
```
用户在每个节点的权力：随时中断、随时发新意图、随时回退

#### 模板 B：单步写操作

```
触发 → 意图被理解
  → 系统组装操作审批场景
  → 呈现Agent的操作草稿（diff视图 / 预览）
  → 用户审批：
     ├─ 批准 → 执行 → 展示结果
     ├─ 修改后批准 → 用户编辑 → 执行 → 展示结果
     └─ 拒绝 → 不执行，回到之前状态
```
用户在每个节点的权力：审批前可修改任何内容、可取消整个操作

#### 模板 C：多步编排

```
触发 → 意图被理解
  → 系统展示执行计划（N步清单）
  → 用户审批计划：
     ├─ 批准整体计划 → 逐步执行
     │    每步完成后展示中间结果
     │    用户可以：暂停 / 跳过 / 接管手动完成 / 修改后继续
     ├─ 修改计划 → 用户调整步骤 → 重新提交
     └─ 拒绝 → 不执行
  → 全部完成 → 展示汇总结果 + 撤销入口
```
用户在每个节点的权力：审批计划、暂停执行、接管任何步骤、撤销已完成步骤

#### 模板 D：Agent主动推送

```
Agent检测到条件（后台）
  → 推送通知（非侵入式）
  → 用户选择：
     ├─ 点击进入 → 切换到对应场景（复用模板A/B/C）
     ├─ 忽略 → 通知消失/归档
     └─ 关闭通知类型 → 该类通知不再推送
```
用户在每个节点的权力：忽略、进入、永久关闭该类通知

#### 模板 E：异常恢复

```
异常发生（Agent中断 / 网络断 / 意图误解 / 权限不足）
  → 系统保留当前状态
  → 展示异常信息 + 用户可选操作：
     ├─ 重试 → 从断点继续
     ├─ 接管 → 切到手动模式完成
     ├─ 取消 → 撤销所有未完成操作，回到之前状态
     └─ 纠正 → 如果是意图误解，重新表达意图
```
用户在每个节点的权力：始终能回到安全状态

### 2.4 流程模板使用规则

```
□ 每个场景（Section 1.3）标注了它使用哪种流程模板？
□ 同一个场景内是否存在模板切换？（如从只读升级到写操作）如有，切换条件是什么？
```

## 3. 控制权流转模型

### 3.1 分层控制权定义

**控制权不是非此即彼——在Agent产品中，不同层级可以同时处于不同控制状态。**

#### 系统级

| 状态 | 含义 | 用户感知信号 |
|------|------|------------|
| 用户浏览 | 用户主动操作，Agent待命 | {描述用户如何感知} |
| Agent执行中 | Agent正在执行任务 | {描述} |
| 等待审批 | Agent暂停等用户确认 | {描述} |
| 降级模式 | Agent不可用，纯GUI | {描述} |

#### 场景级

| 状态 | 含义 | 用户感知信号 |
|------|------|------------|
| Agent场景 | 屏幕内容由Agent动态组装 | {描述} |
| 原生页面 | 传统CRM页面 | {描述} |
| 混合 | 原生页面上有Agent增强区域 | {描述} |

#### 字段级（Agent产品特有）

| 状态 | 含义 | 用户感知信号 |
|------|------|------------|
| Agent自动写入 | Agent已填入值，用户可覆盖 | {描述：如浅色背景+小图标} |
| 等待确认 | Agent建议了值，等用户确认 | {描述：如diff高亮} |
| 用户手动 | 用户自己填写，Agent不介入 | {描述：正常表单字段} |
| 只读 | 系统值，不可编辑 | {描述：灰色} |

### 3.2 状态转移条件

**系统级状态转移：**

| 从 | 到 | 触发条件 |
|----|---|---------|
| 用户浏览 | Agent执行中 | 用户发出操作意图 |
| Agent执行中 | 等待审批 | Agent遇到需确认的字段/操作 |
| 等待审批 | Agent执行中 | 用户批准 |
| 等待审批 | 用户浏览 | 用户拒绝/取消 |
| 任意 | 降级模式 | Agent服务不可用 |
| 降级模式 | 用户浏览 | Agent服务恢复 |

**字段级状态转移：**

| 从 | 到 | 触发条件 |
|----|---|---------|
| 空 | Agent自动写入 | Agent置信度 > 阈值 + 字段为辅助类 |
| 空 | 等待确认 | 字段为核心类，或Agent置信度 < 阈值 |
| Agent自动写入 | 用户手动 | 用户覆盖Agent值 |
| 等待确认 | 用户手动 | 用户修改Agent建议 |
| 等待确认 | Agent自动写入 | 用户确认Agent建议 |

### 3.3 跨层联动规则

## 5. 系统降级策略

### 5.1 降级级别

| 级别 | 触发条件 | 用户能做什么 | 用户不能做什么 |
|------|---------|-----------|-------------|
| L0 · 正常 | Agent服务全部正常 | 所有功能 | — |
| L1 · 部分受限 | {如：意图解析可用但执行引擎不可用} | {列出可用功能} | {列出不可用功能} |
| L2 · Agent不可用 | Agent服务完全不可用 | {CRM所有传统功能正常} | {所有Agent增强功能} |
| L3 · 用户主动关闭 | 用户选择关闭Agent功能 | {同L2} | {同L2} |

### 5.2 降级切换机制

| 从 | 到 | 触发 | 用户感知 | 自动/手动 |
|----|---|------|---------|----------|
| L0 | L1 | {如：MCP断连} | {如：提示"部分AI功能暂不可用"} | 自动 |
| L0 | L2 | {如：Agent服务宕机} | {如：提示"AI助手暂时离线，CRM正常使用"} | 自动 |
| L0 | L3 | 用户主动关闭 | {如：设置页面开关} | 手动 |
| L1/L2 | L0 | {如：服务恢复} | {如：提示"AI助手已恢复"} | 自动 |
| L3 | L0 | 用户重新开启 | {如：设置页面开关} | 手动 |

### 5.3 降级状态的体验底线

```
□ L2状态下，CRM所有核心业务流程（创建客户/跟进/商机管理）是否100%可用？
□ Agent执行到一半服务中断时，已完成的操作是否保留？未完成的是否安全回滚？
□ 用户是否始终知道当前系统处于什么降级级别？
□ 从降级恢复时，是否有"恢复确认"而非静默恢复？
```
