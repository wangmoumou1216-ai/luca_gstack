# AI Native 品味锚点（4 + 3 体系）

> 共享 reference。被 /taste-review、/design-brief、/ux-brainstorm、/design-review 引用。
> 本文件用 **诊断 + 处方** 双栏结构。不只告诉你"什么是不好的"，
> 还告诉你"具体怎么做能通过"。
> 每个处方按"低/中/高实现成本"分级，配 B2B 销售场景的具体示例。
> 结构方法论借鉴：Microsoft HAX Design Patterns 的 Problem/Solution/When to use/How to
> use/Benefits/Pitfalls 六段式。

---

## 0. 为什么要升级

原有四锚点全部围绕**效率**：入口少、路径短、层次清、AI
入口在上下文内。这是 2019-2022 年 SaaS 最佳实践。

AI Native 产品（2023+）新增了两个维度，原锚点无法覆盖：

1. **信任维度** — AI 输出的不确定性、来源、推理是否可追溯。
2. **代理维度** — AI 代替用户做事时，用户如何始终保持控制权。

`Ryo Lu / Linear / Attio / Notion / Raycast` 作为效率锚点继续有效；
`Perplexity / Cursor / Granola` 补齐信任与代理锚点。

## 1. 效率类锚点（原有 4+1，保留）

---

### 锚点 1 · Ryo Lu — 任务对齐（每个 UI 元素必须服务具体任务）

**定位：** 每个 UI 元素必须能回答「它在帮用户完成什么任务」。
任何"展示功能强大"、"让界面看起来更专业"的元素都是污染。

#### 诊断信号（不通过的征兆）

- 主动打开页面后，有元素让你想"这个是干嘛的？"
- 元素存在的理由是"为了让界面看起来更完整 / 更像产品"
- 元素可以直接删除，整个任务流程不受影响
- 元素的用途只能用"展示 AI 能力"、"说明我们很智能"这类抽象语句描述

#### 实现方式（按成本分级）

**方式 A · 删除（零成本，最推荐）**
- 直接删掉这个元素
- 不要改成小号版本，不要折叠，不要放到二级菜单——删除
- 删除后观察：任务流程变难了吗？没变难 → 证明该删

**方式 B · 重定位（低成本）**
- 如果元素有些许价值但位置错了（占据了黄金区域但只服务次要任务），把它挪到角落
- 比如：主列表页右上角的"AI 摘要"卡片 → 挪到详情页右侧辅助区

**方式 C · 合并（中成本）**
- 如果元素有价值但独立存在是累赘，合并到已有组件里
- 比如：独立的"AI 建议条" → 合并到每行客户记录的悬停菜单里

#### B2B 销售场景反例 vs 正例

```
❌ 反例 · 客户列表页顶部的"AI 洞察卡片"
  - 显示："本周潜在成单客户增加 12%"
  - 销售为什么不用：信息没颗粒度，点不进去看详情，每天看都一样
  - 处方：删除（方式 A）

✅ 正例 · 客户列表行内的"下次跟进建议"徽章
  - 显示："建议 3 天内联系" + hover 显示依据
  - 销售为什么用：直接告诉我下一步，hover 能追溯，省去判断
```

#### 不通过动作

**删除该元素。** 不重新设计，不缩小，不折叠——直接删除。超过 30% 的
UI 元素无法回答"帮用户完成什么任务"时，整个页面要返工到 design-brief
阶段。

### 锚点 2 · Linear — 路径效率（核心任务 ≤ 3 步 + 即时反馈）

**定位：** 核心任务路径 ≤ 3 步。操作完成后结果即时可见，无需刷新、
无需返回、无需等待异步通知。

#### 诊断信号

- P0 任务从触发到完成 > 3 步
- 每步之间用户要等待（> 500ms 且无进度指示）
- 操作完成的反馈不在用户注意力所在位置（顶部 toast、另一页、邮件通知）
- 操作需要用户主动刷新才能看到结果

#### 实现方式（按成本分级）

**方式 A · 合并步骤（低成本）**
- 把"选择 + 确认"合并成单点操作（Optimistic UI）
- 把"填表单 + 提交"改成"inline editing"（点击字段直接编辑，离焦保存）
- 典型：把"打开表单 → 填写 → 保存 → 返回列表" 4 步 →
  "列表行内点击字段直接编辑" 1 步

**方式 B · 异步改乐观 UI（中成本）**
- 保留异步后端但前端立即给结果
- 用户点"删除" → 立即从列表消失（UI 乐观） → 后端失败时弹回并解释
- 风险管理：错误率 < 5% 的场景才适用，不要在涉及钱的场景用

**方式 C · 并行执行（高成本）**
- 把串行步骤改成并行
- 典型：同时加载"客户基本信息"和"历史跟进记录"，不要等前者完成再请求后者

#### B2B 销售场景反例 vs 正例

```
❌ 反例 · 推进商机阶段
  1. 列表页点商机
  2. 详情页滚动找"阶段"字段
  3. 点编辑
  4. 下拉菜单选新阶段
  5. 点保存
  6. 看到弹窗"保存成功"
  7. 返回列表看变化
  → 7 步，远超 3 步底线

✅ 正例 · Linear 式推进
  1. 列表行 hover 显示阶段徽章可点击
  2. 点击后下拉切阶段
  3. 立即变更（乐观 UI），无需跳转
  → 2 步
```

#### 不通过动作

找可消除的步骤；合并两步；异步改乐观 UI。超 3 步但无法简化时，要在
design-brief 记录"已知违反 Linear 锚点" + 合理理由。

### 锚点 3 · Attio — 密度有层次（L1/L2/L3 三层可辨认）

**定位：** 信息密度高但有层次。B2B 销售用户需要密度（一屏看更多信息 =
更少翻页 = 更高效），但密度不能是意外的——用户要在 3
秒内知道什么最重要。

#### 诊断信号

- 3 秒内说不出页面上"最重要的信息"
- L1 和 L2 的视觉区分不稳定（这个页面 L2 比 L3 大，另一页反过来）
- 用留白和分割线制造层次，而不是字号/字重/颜色
- 所有字段用同一个字号 + 同一个颜色

#### 实现方式（按成本分级）

**方式 A · 四级字号规范（低成本，纷享销客规范）**
```
L1 区块标题 / 最重要数据：text-15 font-medium text-n19（15px/500/#181C25）
L2 正文 / 字段值：text-13 font-normal text-n19（13px/400/#181C25）
L3 字段标签 / 次要说明：text-13 font-normal text-n11（13px/400/#91959E）
L4 辅助 / 时间戳：text-12 font-normal text-n11（12px/400/#91959E）
```
适用：所有 B2B SaaS 信息密集场景

**方式 B · 颜色 + 字重分层（中成本）**
- L1 用品牌色或深黑（#181C25），吸引视线
- L2 用中黑（#181C25）但字重 400
- L3 用浅灰（#91959E）主动降权
- 同行信息层级：L1 字段名 + L2 字段值 一行显示

**方式 C · 空间 + 对齐分层（高成本，精细化）**
- L1 独占一行，左对齐
- L2 紧跟 L1，缩进 4-8px
- L3 右对齐或灰色小字填充
- 用格子对齐而非自由流

#### B2B 销售场景反例 vs 正例

```
❌ 反例 · 客户详情页所有字段都用 14px / #333
  - 客户名、成立时间、行业、电话、地址 同等权重
  - 销售扫视时视线到处跳，找不到焦点

✅ 正例 · 四级字号分层
  - L1：客户名 15px/500 + 行业标签（紧跟）
  - L2：核心联系人 + 上次跟进时间（13px/400 黑）
  - L3：地址、注册时间（13px/400 灰）
  - L4：创建时间、最后更新（12px/400 灰）
```

#### 不通过动作

重新规划信息层级；不要靠加空白/加卡片/加分割线美化。对照
`html-prototype-tokens.md` 第 2 节的四级字号规范重做。

### 锚点 4 · Notion — 入口克制（首屏主操作 ≤ 3 个）

**定位：** 首屏主要操作入口 ≤ 3 个，无重复入口。

#### 诊断信号

- 首屏 800px 高度内可见主操作 > 3 个
- 同一操作有多个入口（顶部按钮 + 右键菜单 + 侧栏 + 卡片操作条）
- 次要操作平铺暴露，没进 overflow menu

#### 实现方式（按成本分级）

**方式 A · 合并重复入口（低成本）**
- 同一操作只保留 1 个入口，其他删除
- 规则：哪个入口使用频次最高，留哪个

**方式 B · Overflow menu 归类（中成本）**
- 低频操作放进"更多"（三个点菜单）
- 页面首屏只保留 P0 操作按钮

**方式 C · 上下文菜单（高成本）**
- 根据选中对象动态显示可用操作
- 比如列表页没选行时只显示"新建"；选了行才显示"删除/导出/批量操作"

#### B2B 销售场景反例 vs 正例

```
❌ 反例 · 客户列表页首屏
  - 顶部：新建客户 / 导入 / 导出 / 批量操作 / 筛选 / 视图切换
  - 每行：编辑 / 删除 / 标记跟进 / 分配
  → 首屏主操作 10+ 个

✅ 正例
  - 顶部：新建客户（主按钮）+ "更多"（下拉含导入/导出/批量）
  - 每行默认不显示操作，hover 时显示"编辑 / 标记"
  - 点击行进入详情后，其他操作在详情页显示
```

#### 不通过动作

重新分组合并；低频操作塞到 overflow menu；同类操作合并成下拉。

### 锚点 5 · Raycast — AI 入口位置（在任务上下文内）

**定位：** AI 入口必须在任务上下文内；AI 介入后路径只能变短或持平，不能变长。

#### 诊断信号

- AI 入口是独立侧栏 / 浮球 / 新标签页
- 用户要用 AI 必须跳出当前任务
- 用 AI 的路径比不用 AI 还长
- AI 入口的文案是"AI 助手"、"智能问答"等抽象名词（不告诉用户点了会发生什么）

#### 实现方式（按成本分级）

**方式 A · 嵌入字段旁（低成本）**
- 跟进记录表单的每个字段右侧加 ✨ 图标：点击由 AI 补全
- 触发成本：1 次点击
- 失败 fallback：手动继续填

**方式 B · 选中触发（中成本）**
- 用户选中表格行 → 浮出菜单包含 "AI 分析这批客户"
- 用户选中文字 → 浮出菜单包含"AI 改写 / 翻译 / 摘要"
- 触发成本：0 次点击（选中自动出现）

**方式 C · Cmd+K Palette（高成本）**
- 全局快捷键唤起命令面板
- 输入自然语言即可触发任意任务
- 适用于高级用户 / 有良好 onboarding

#### B2B 销售场景反例 vs 正例

```
❌ 反例 · 独立的"AI 助手"浮球
  - 固定在右下角，点击展开聊天面板
  - 销售在填跟进记录时想让 AI 帮忙 → 跳出表单去聊天 → 复制 AI 回答 → 回表单粘贴
  - 路径：8 步，远超直接手写的 3 步

✅ 正例 · 跟进记录字段内的 Tab 补全
  - 销售输入前几个字："今天和李总通话..."
  - AI ghost text 显示灰色预测："...讨论了 Q2 采购计划，对方关注交付周期"
  - 按 Tab 接受 / 继续打字拒绝
  - 路径：1 步
```

#### 不通过动作

**独立 AI 入口 → 嵌入到具体任务上下文。** **AI 路径变长 → 移除
AI**（此场景不适合 AI）。

## 2. 信任与代理类锚点（新增 3 个）

---

### 锚点 6 · Perplexity — 可追溯（AI 输出必须有依据）

**定位：** AI 输出必须**可追溯**。用户能看到 AI 凭什么得出这个结论，并能点击验证。

#### 诊断信号

- AI 给出的摘要 / 洞察 / 建议 / 预测，没有任何来源标注
- AI 输出用肯定句呈现，没有置信度或 hedging 语言
- 用户点不进去查依据
- AI 推理过程完全不可见（不能展开看）

#### 实现方式（按成本分级）

**方式 A · Inline Citation（低成本，最推荐）**

适用：任何有具体数据源的 AI 输出。

做法：结论后加可点击的角标，hover 显示来源摘要，点击跳转原始数据。

```html
<div class="text-13">
  预测成单概率 73%
  <sup class="text-12 text-primary cursor-pointer" title="基于近 30 天 47 个同类客户的历史数据">
    [1]
  </sup>
</div>
```

B2B 场景示例：
- "跟进建议：3 天内联系 [基于该客户最近 2 次成交周期 14 天]"
- "风险提示：预计流失 [基于该客户 30 天无登录 + 续费倒计时 < 45 天]"

**方式 B · Confidence Indicator（中成本）**

适用：分类、评分、风险预警类输出。

做法：高/中/低标签 + 颜色 + 一句话说明"什么数据支撑了这个判断"。

```html
<div class="flex items-center gap-2 text-12">
  <span class="text-n11">AI 分析：</span>
  <span class="bg-warning-bg text-warning border border-warning-border px-2 py-0.5 rounded">
    置信度：中
  </span>
  <span class="text-n11">依据：2 次有效联系 + 1 次报价</span>
</div>
```

B2B 场景示例：
- 客户重要性评级（高/中/低 + 依据）
- 商机紧急度（紧急/标准/低 + 依据）

**方式 C · Counterfactual Explanation（高成本，高价值场景）**

适用：重大决策类输出（合同、客户分级、大额商机）。

做法：不只给结论，同时给"如果 X 变了，结论会变成 Y"的敏感性说明。

```html
<div class="bg-special01 p-3 rounded text-13 text-n15">
  <div class="font-medium text-n19 mb-1">建议优先级：第 1 位</div>
  <div class="text-12 text-n11">
    如果合同截止日期推迟 2 周 → 优先级降至第 3 位<br>
    如果对方追加需求 → 优先级保持第 1 位
  </div>
</div>
```

B2B 场景示例：
- 合同风险预测 + "如果条款 A 修改则降低风险"
- 客户流失预警 + "如果下周联系则降低流失概率 40%"

#### B2B 销售场景反例 vs 正例

```
❌ 反例 · "本周 AI 洞察：您的成单率可能下降"
  - 点不进去
  - 没依据
  - 没置信度
  - 典型 AI Slop

✅ 正例 · 周报卡片里的"本周洞察"
  - "成单率从 22% → 18%（↓4%）[1]"
  - 点击 [1] 展开：基于 47 个商机的阶段进展 + 对比 4 周前的 22%
  - 附带"置信度：高 | 样本量充足"
```

#### 不通过动作

**阻断级锚点，一票否决。**
- 无来源的 AI 输出：要么加来源（方式 A），要么移除这个 AI 输出（不能裸展示）
- 无不确定性表达的 AI 输出：按 `ai-native-design-framework.md` 第 5 节选一种方式补齐

### 锚点 7 · Granola — 诚实表达（AI 不装作确定）

**定位：** AI 不装作确定。在模型能力边界附近，
产品必须诚实地表达"我不太确定"、"这是初步的"、"可能有误"。

#### 诊断信号

- AI 输出用肯定句表达推测（如 "该客户会流失"）
- 新用户 / 数据不足时 AI 硬给结果（不说"数据不够"）
- 超出 AI 能力的请求硬生成答案（不说"我不太确定"）
- AI 输出的语气和 confidence 无关（低置信也用自信语气）

#### 实现方式（按成本分级）

**方式 A · 语言 Hedging（零成本）**

在 AI 输出的文案里加不确定性语言：
- 肯定 → "看起来可能会"
- "将会" → "建议考虑"
- "是" → "根据现有数据判断是"
- "应该" → "初步判断应该"

```
❌ AI 输出："该客户将在 30 天内流失。"
✅ AI 输出："根据近期行为，该客户可能在 30 天内流失。建议本周内联系。"
```

**方式 B · 冷启动主动告知（低成本）**

当 AI 输入数据不足时，UI 主动告知而不是硬生成：

```html
<div class="bg-special01 border border-n05 rounded p-4">
  <div class="text-13 text-n15">
    暂无足够数据生成 AI 建议（该客户仅有 1 次联系记录）
  </div>
  <div class="text-12 text-n11 mt-2">
    建议在 2 次联系后再使用 AI 分析。
  </div>
</div>
```

**方式 C · 拒答态 + 替代操作（中成本，详见 `html-prototype-tokens.md` 第 4 节）**

超出能力时明确拒答，同时给替代路径：

```html
<div class="bg-warning-bg border border-warning-border rounded p-4">
  <div class="text-13 font-medium text-warning mb-1">
    AI 暂时无法给出可靠建议
  </div>
  <div class="text-13 text-n15 mb-3">
    该请求涉及跨地区法务合规，需要专家人工判断。
  </div>
  <div class="flex gap-3">
    <button class="text-13 text-primary hover:underline">查看类似案例</button>
    <button class="text-13 text-primary hover:underline">联系法务支持</button>
  </div>
</div>
```

#### B2B 销售场景反例 vs 正例

```
❌ 反例 · 对新客户（只 1 次联系）的预测
  - UI："成单概率 73%"
  - 问题：1 次数据不足以支撑这个数字，但 UI 给了精确到个位的数字
  - 销售信了 → 投入时间 → 失败 → 对 AI 失去信任

✅ 正例 · 同样的场景
  - UI："数据不足 | 建议积累 3+ 次联系后再评估"
  - 次级 UI："基于行业平均值，此类客户成单周期约 60-90 天 [行业数据]"
  - 销售得到真实信号，做出正确判断
```

#### 不通过动作

**阻断级锚点。B2B 用户对虚假自信零容忍。**
- 过度自信 → 方式 A 加 hedging
- 冷启动硬给结果 → 方式 B 加"数据不足"态
- 超边界硬生成 → 方式 C 加拒答态

## 3. 代理维度（1 锚点，涉及 agent 动作 / 场景 D 强制）

---

### 锚点 8 · Cursor — 用户控制权（Agent 不是黑箱）

**定位：** 当 AI 代替用户做事时，用户必须始终能**看见、暂停、接管、
撤销**。Agent 不是黑箱。

#### 诊断信号

- Agent 执行过程不可见（只看到最终结果 + 一个"已完成"）
- 无法暂停正在执行的 Agent
- 想接管只能"整个重来"
- Agent 动作不可撤销（发了邮件就发了，没 undo）
- 对外通讯（邮件/短信）默认直发而不是草稿

#### 实现方式（按"可见/暂停/接管/撤销"四项分别说明）

**可见性 — Plan Preview + Step Stream**

**低成本：** 执行前显示计划清单，用户批准后整体执行。

```html
<div class="bg-info-blue-bg border border-info-blue-border rounded p-4">
  <div class="text-13 font-medium mb-2">AI 即将执行以下 3 步：</div>
  <ol class="space-y-1 text-13 text-n15 mb-3">
    <li>1. 查询近 30 天的跟进记录</li>
    <li>2. 按成单概率排序客户</li>
    <li>3. 生成本周跟进计划并保存到"我的任务"</li>
  </ol>
  <button class="text-primary">批准并执行</button>
  <button class="text-n15 ml-3">取消</button>
</div>
```

**中成本：** 执行中实时展示每步进度。

```html
<div class="space-y-2">
  <div class="flex items-center gap-3 text-13">
    <span class="w-5 h-5 rounded-full bg-info-green text-white flex items-center justify-center">✓</span>
    <span>已查询 47 条跟进记录</span>
  </div>
  <div class="flex items-center gap-3 text-13">
    <span class="w-5 h-5 rounded-full bg-info-blue animate-pulse"></span>
    <span>正在按成单概率排序...</span>
  </div>
  <div class="flex items-center gap-3 text-13 text-n11">
    <span class="w-5 h-5 rounded-full border-2 border-n07"></span>
    <span>生成计划</span>
  </div>
</div>
```

**暂停性 — Pause + Persist State**

执行中随时可暂停，状态保留。

```html
<div class="flex items-center gap-3">
  <div class="flex-1">AI 正在处理 (2/5)</div>
  <button class="text-13 text-n15 hover:text-n19">暂停</button>
</div>
<!-- 暂停后 -->
<div class="bg-warning-bg p-3 rounded">
  <div class="text-13">已暂停在第 3 步。可以随时恢复。</div>
  <div class="flex gap-3 mt-2">
    <button class="text-13 text-primary">恢复</button>
    <button class="text-13 text-n15">接管（手动完成剩余步骤）</button>
  </div>
</div>
```

**接管性 — Hand-off to Manual**

从暂停状态切换到手动，已完成步骤保留。

```html
<div class="space-y-2">
  <div>已完成步骤（Agent 完成）：</div>
  <div class="text-13 text-n15">✓ 查询数据 ✓ 排序客户</div>
  <div class="mt-3">剩余步骤（由您手动完成）：</div>
  <div class="text-13">☐ 生成计划 — <button class="text-primary">开始手动</button></div>
</div>
```

**可撤销性 — Undo Receipt**

每个 Agent 动作完成后给"回执"，支持撤销。

```html
<div class="bg-special01 p-3 rounded">
  <div class="text-13">AI 已完成：</div>
  <ul class="text-12 text-n11 mt-1 space-y-1">
    <li>修改了 5 条商机阶段（<a class="text-primary">查看</a>）</li>
    <li>生成了 1 份周报（<a class="text-primary">查看</a>）</li>
    <li>草拟了 3 封跟进邮件（<a class="text-primary">查看</a>，未发送）</li>
  </ul>
  <button class="text-13 text-primary mt-2">全部撤销</button>
</div>
```

**对外通讯强制规则：默认草稿**

涉及邮件/短信/群消息的 Agent 动作：
- ❌ 绝不默认直发
- ✅ 生成草稿 + 明确提示"已放入草稿箱，请审核后发送"
- ✅ 需要用户点击"发送"才真正发出

#### B2B 销售场景反例 vs 正例

```
❌ 反例 · "一键整理本周线索"
  - 用户点击按钮
  - 几秒后通知"已完成：整理了 47 条线索"
  - 用户不知道：AI 改了哪些？按什么规则？涉及哪些客户？能不能撤销？
  - 后果：一周后发现分类错了，手动恢复 3 小时

✅ 正例 · 同样场景
  - 用户点击按钮
  - 显示：计划清单（3 步）+ 预计影响 47 条数据
  - 用户批准 → 执行中显示每步进度
  - 完成后给回执 + 撤销按钮（7 天内可撤销）
  - 对外通讯类动作（通知客户）默认草稿
```

#### 不通过动作

**阻断级锚点。任意一项不通过，agent 功能不能上线。**
- 不可见 → 加 Plan Preview 或 Step Stream
- 不可暂停 → 加 Pause 机制
- 不可接管 → 加 Hand-off 路径
- 不可撤销 → 加 Undo Receipt
- 对外通讯默认直发 → 强制改为草稿模式

任意一项不通过，返回 `/design-brief` Phase 1 Layer D 重新设计。
