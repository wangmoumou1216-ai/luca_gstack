## 语义组（6 主题，结论可入 Packet）

### 1. 有限状态建模（State Machine）`[语义可入 Packet]`

把复杂 UI 行为建成有限状态机：**状态**（UI 所处的互斥模式）、**事件**（触发迁移的动作/超时/响应）、**迁移规则**（在状态 A 收到事件 X → 状态 B）、**守卫**（迁移前置条件，如 isValid）、**进出动作**（副作用）。
- 建模步骤：列全状态 → 列全事件 → 定义合法迁移 → **点名要杜绝的不可能状态**（如 loading 与 error 并存）→ 加守卫 → 每状态映射一个 UI 表现。
- 常用骨架：表单 `idle→editing→validating→submitting→success/error→idle`；取数 `idle→loading→success/error→retrying`；多步向导 `step1→…→review→submitting→complete`。
- 铁律：每个状态都要有出路（无死端）；先建 happy path 再补错误态；一个状态机管一件事。
- **衔接点**：这是 `references/ai-native-state-coverage.md` 12 态模型的建模方法层——12 态定义"要覆盖哪些状态"，本主题定义"状态之间怎么迁移、哪些组合不可能"。ux-brainstorm 交互架构文档 §3.2 的状态流转表（从/到/触发条件）即此方法的落点。
- Packet 句式示例：「批量导入流程必须杜绝 `submitting 与 editing 并存`；失败后必须可从 error 单步回到 editing 且保留已填数据」。

### 2. 错误恢复三段（Prevention → Detection → Recovery）`[语义可入 Packet]`

错误体验分三层设防，每层都是可声明的语义要求：
- **预防**：约束式输入（选择器代替自由文本）、智能默认值、破坏性操作二次确认、自动保存防丢失。
- **检测**：字段级实时校验 + 提交时表单级校验 + 网络/超时/权限检测。
- **恢复**：出错**不清空用户已填内容**；瞬时故障给重试（网络错误自动退避重试）；给替代路径；误操作给撤销。
- 错误信息内容语义三要素：**何事**（发生了什么，人话不是错误码）+ **为何**（有助理解才写）+ **怎么办**（具体下一步）。永不指责用户；不写"Something went wrong"这类空话。
- **衔接点**：state-coverage 状态 3（错误态"避免冷冰冰的 Error"）/状态 4（部分完成态 retry/skip/manual）的上游方法——那里定义状态该长什么样，这里定义整条防线的分层职责。
- Packet 句式示例：「所有录入型操作出错后必须保留用户输入并提供单击重试；权限类失败必须说明缺什么权限、向谁获取」。

### 3. 决策与分块（Hicks + Miller/Cowan）`[语义可入 Packet]`

- **Hicks**：决策时间随选项数对数增长（`RT = a + b·log₂(n+1)`）。要点不是砍功能，是**控制同屏同时呈现的选择数**：分组化 > 删除；高频动作给默认值直接跳过选择；渐进披露（先默认后高级）。选项**质量**与数量同权重——五个清晰互斥的选项好过三个含混的。
- **Miller 修正**：别引用"7±2"当设计规则——Cowan (2001) 研究指出工作记忆的**有意义分块**上限约 **4±1**。可操作的是分块策略本身：导航按类分组（3 组 ×3-4 项好过 10+ 平铺）、长表单切成可独立完成的分节、进度用 3-5 个命名阶段而非"第 2/12 步"、熟练用户分块容量大于新手。
- **衔接点**：`ai-native-taste-anchors.md` Notion 锚点「首屏主要操作入口 ≤ 3 个」是 Hicks 在本框架的既有落点；本主题给出它的机理与延伸（分组/默认值/渐进披露）。
- Packet 句式示例：「工作台首屏同时呈现的主操作不超过 3 个，次级操作收进分组入口；批量操作向导每步只呈现一个决策」。

### 4. 系统响应时序（Doherty Threshold）`[语义可入 Packet]`

Doherty & Thadani (IBM, 1982)：系统响应 **<400ms** 时用户保持心流，超过则注意力开始漂移。分级表：

| 响应时间 | 感知 | 必须达成的语义 |
|---|---|---|
| 0-100ms | 即时 | 按压反馈必须在 100ms 内出现（无论底层动作多慢） |
| 100-400ms | 快 | 无需 loading 指示——**<400ms 完成的动作闪现 spinner 本身就是干扰** |
| 400ms-1s | 可感等待 | 必须有响应指示 |
| 1-3s | 明确等待 | loading 态必须出现 |
| 3s+ | 心流已断 | 必须给进度（非裸 spinner），可考虑乐观 UI |
| 10s+ | 任务级中断 | 详细进度 + 预估时间 + 允许转后台 |

- 高频交互优先分配时延预算（搜索、筛选、切换视图）；自动补全建议 300ms 内出首批。
- **概念区分（重要）**：`html-prototype-tokens.md` §7 的 `--duration-max: 400ms` 是**动效时长上限**（转场动画不得超过 400ms）；本主题是**系统响应等待阈值**（用户等系统的时间）。同一个数字、两个不同概念——一个刻意的 250ms 入场动画完全合法，它不是"等待"。
- Packet 句式示例：「记录检索必须在 400ms 内出现首批结果或等待指示；所有按钮按压必须 100ms 内有视觉反馈」。

### 5. 搜索语义（Search UX）`[语义可入 Packet]`

CRM 类产品记录查找是核心交互，可声明的语义要求：
- **入口**：placeholder 说明可搜什么（"搜索客户、商机或联系人"而非"搜索…"）；输入后出现清除按钮；查询词在结果中保留可改（免重打）。
- **建议**：2-3 字符后出建议；顺序=最近搜索→热门→预测补全；建议内高亮已输入部分；上限 5-8 条（Hicks）。
- **导航式搜索**：用户会用搜索当导航（搜"设置""发票"）——目标页面存在时必须直达，不得报零结果。
- **零结果态（最关键、最易被忽视）**：确认搜了什么（含错拼提示"未找到 'onbording'"）→ 给纠错/更宽词建议 → 给替代出路（浏览分类/联系支持）。**零结果 = UX 失败信号**，永不留白页。
- **搜索日志是最诚实的用户反馈**：高频查询揭示导航缺口，零结果查询揭示内容缺口。
- **衔接点**：state-coverage 状态 2（空态"必须有提示+引导操作"）在搜索场景的专化；查询容错（错拼/同义/单复数）是工程要求，写进 tech-spec 而非 Packet。
- Packet 句式示例：「全局搜索必须支持导航式意图直达；零结果态必须含纠错建议与浏览出路」。

### 6. 表单校验时机（Form Feedback Timing）`[语义可入 Packet]`

- **校验时机 = on blur**（离开字段时），不是每次击键——键入中实时报错是干扰；也不是只在提交时——那让用户批量返工。
- 错误就地显示在字段正下方（不是表单顶部）；多字段服务端错误：能定位到字段就 inline，多处受影响才在顶部汇总（且汇总可键盘聚焦、逐条链到字段）。
- 正确性不自明的字段（密码强度/用户名可用性）给成功指示；有字数限制就始终显示剩余数。
- 多步表单语义：每步是可独立完成的单元；可回退不丢数据；长表单自动保存；放弃部分输入前必须确认。
- **衔接点**：`design-system-contract.md` §四（表单字段 24px 垂直间距）/§七（footer 取消左、主操作右）管表单的**视觉规范**；本主题管**反馈时机与错误语义**——两者互补不重叠。
- Packet 句式示例：「所有表单字段离焦即校验；错误信息必须指出具体修复方式（'邮箱需包含 @' 而非 '格式错误'）；多步录入可回退且不丢已填数据」。

## 仅供推理组（3 主题，禁入一切下游交付物）

### 7. 布局处方（表单单列/标签顶对齐等）`[仅供推理]`

单列几乎总是对的（双列打乱阅读流与填写顺序）、标签顶对齐扫读最快、字段宽度暗示预期输入长度、"标可选不标必填"减少视觉噪音、能删的可选字段都删掉（字段越少完成率越高）。
**为什么禁入**：这些是 OD/所选 design system 的版式执行层决定。用途：ux-brainstorm 方案推理时判断"这个流程的表单复杂度是否失控"、评审 html-prototype 本地产物时当检查项。

### 8. 目标尺寸与距离（Fitts）`[仅供推理]`

`MT = a + b·log₂(2D/W)`——大目标+近距离=快而准。触屏最小 44×44pt（Apple HIG）/48×48dp（Material）；指针最小 24×24px；**命中区 ≠ 视觉尺寸**（16px 图标可配 44px 命中区，用 padding 扩）；屏幕边角是无限大目标（指针不会过冲）；相关操作放在其作用对象近旁；破坏性操作反用——做小放远防误触。
**为什么禁入**：px/pt 值与位置指令是执行层。用途：推理"高频操作的可达性是否成立"、审查本地原型的误触风险（高错误率交互几乎都是 Fitts 失败）。

### 9. 加载呈现选型（skeleton vs spinner vs 乐观 UI）`[仅供推理]`

已知布局的内容用骨架屏（保持空间上下文，感知更快）；未知时长用 spinner（小而不抢戏）、可测进度用进度条；低风险写操作用乐观 UI（立即显示预期结果，失败回滚）；列表分批入场（30-50ms 间隔）；内容加载不得引起布局跳动；shimmer 动画尊重 reduced-motion。
**为什么禁入**：控件选型是 OD/DS 的执行层决定。**时序语义归主题 4**（何时必须出现 loading/进度是可入 Packet 的）；选型本身只用于推理与本地路径。
**衔接点**：`html-prototype-tokens.md` §4.2 已有 skeleton 模板、`ai-native-taste-anchors.md` Linear 锚点方式 B（异步改乐观 UI）——本地生成路径的既有落点。
