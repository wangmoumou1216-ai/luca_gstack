# AI Native 设计方法论

> 共享 reference。被 /ux-brainstorm、/design-brief、/ux-research、/redteam 引用。
> 任何 skill 在做"AI 原生"相关判断时，以本文件为准，不各自重新定义。

---

## 0. 判定红线：什么不是 AI Native

在任何 AI 相关的设计决策开始前，先排除以下三类伪 AI Native：

1. **加按钮派：** 在现有界面上加一个「AI 助手」入口/浮球/侧栏，
   核心任务路径不变 → **不是 AI Native**。
2. **换壳派：** 把一个现有筛选器改成自然语言输入框，
   但背后还是同样的结构化查询 → **不是 AI
   Native**（除非自然语言能表达结构化查询表达不了的意图）。
3. **贴图派：** 在结果边上加一段 AI 生成的"摘要"、"洞察"、"建议"，
   但不影响用户下一步决策 → **不是 AI Native**。

**AI Native 的唯一判定标准：**
> 引入 AI 之后，用户**完成任务的决策路径、输入方式、
> 或审核模式**是否被**结构性**地重构？
>
> 如果拿掉 AI，这个功能的信息架构和主路径完全不变 → 它不是 AI Native，是 AI 装饰。

## 1. AI 原生四范式（选型用）

任何 AI Native 设计决策必须先回答一个问题：**本次用的是哪个范式？**
不允许回答"综合"或"都有一点"——这是分析不清楚的信号。

### 范式 A：对话式替代（Conversational Replacement）

**定义：** 用自然语言对话替代表单 / 筛选 / 菜单操作。用户说出意图，系统直接执行。

**典型场景：** 复杂筛选、跨对象查询、多参数配置。

**标杆：** Perplexity（自然语言检索替代 Google 高级搜索）、Raycast AI Commands、
Linear 的 `Command + K` AI 语义搜索。

**成立前提：**
- 用户意图能被自然语言清晰表达（否则结构化表单更快）
- 系统对意图的解析准确率 ≥ 用户可接受阈值（B2B 场景通常 > 90%）
- 有显式的 fallback：解析不了时退回结构化

**在 B2B 销售场景的适用性：**
- 高 — 复杂线索筛选、跨时段业绩查询、客户活动回溯
- 低 — 高频重复录入（语音/文字输入比选择慢）、强结构化数据（如金额、日期）

---

### 范式 B：嵌入式预测（Inline Prediction）

**定义：** 在用户操作上下文中实时预测下一步，不提供独立 AI 页面。AI
是路径的一部分，不是路径的入口。

**典型场景：** 智能补全、预填默认值、字段联动推断、下一步操作建议。

**标杆：** Cursor（Tab 补全代码）、Granola（录音时实时转录 + 关键词标记）、
Notion AI（行内 `/` 触发）、Gmail Smart Compose。

**成立前提：**
- 预测发生在用户注意力所在的位置（不是另起一块）
- 用户可以无缝忽略预测（继续原操作路径不受干扰）
- 预测接受成本 ≤ 手动完成成本（否则预测没意义）

**在 B2B 销售场景的适用性：**
- **最高** — 跟进记录补全、客户画像自动填充、下一步动作建议、商机阶段推进预测
- 这是 B2B SaaS 最容易出效果的范式，优先考虑

---

### 范式 C：代理式执行（Agentic Execution）

**定义：** AI 替用户完成多步骤任务（自动分类、自动跟进、
自动生成报价、自动安排会议），用户只做最终确认或中途干预。

**典型场景：** 重复性行政操作、需要跨系统协作的任务、基于规则可预测的判断。

**标杆：** Cursor Agent（plan → execute → review）、Granola（会后自动生成纪要 +
推送待办）、Claude Projects（任务规划 + 执行）、Lindy、Devin。

**成立前提：**
- 任务的成功/失败可被明确判定（否则用户审核成本比做还高）
- 有"计划预览"阶段让用户审批（除非已建立信任）
- 有完整的 undo / rollback / 接管机制
- Agent 动作过程可见、可暂停（不是黑箱后台任务）

**在 B2B 销售场景的适用性：**
- 高 — 自动整理线索、自动跟进记录归档、自动生成周报、自动推送客户近期动态
- 风险 — 涉及对外通讯（发邮件给客户、发消息给群）的 agent
  必须默认"草稿模式"，绝不可默认直发

**场景 D 的触发条件：**
当一个功能的设计目标是"从用户主动操作 → 变为用户监督 agent"，
该功能进入**场景 D（Agent 化改造）**，执行路径与 A/B/C 不同，详见各 skill
的场景 D 分支。

---

### 范式 D：决策增强（Decision Augmentation）

**定义：** AI 在关键决策节点提供**结构化建议**（含置信度 + 依据），
不替代用户决策，但降低认知负担。

**典型场景：** 定价建议、优先级排序、风险预警、归因分析。

**标杆：** Perplexity（回答自带 sources）、Attio AI（客户风险评分附依据）、
Linear（issue 优先级建议）、Arc Browser（tab 整理建议）。

**成立前提：**
- 建议附带**可追溯的依据**（引用了哪些数据 / 历史 / 规则）
- 建议附带**置信度或不确定性表达**（高/中/低，或百分比，或 hedging 语言）
- 用户可以一键"按建议执行"也可以"按自己的来"，两条路径平权

**在 B2B 销售场景的适用性：**
- 高 — 客户成单概率预测、最佳跟进时间建议、商机风险预警、合同条款合规检查
- 这个范式对"信任"要求最高：销售把客户关系押在上面，AI 错了要赔客户

## 2. AI 介入判定矩阵（具体场景具体分析）

**不是每个功能都适合 AI Native。** 在决定"是否引入 AI"之前，用以下矩阵做一次显式判断：

| 维度 | 低 AI 倾向 | 高 AI 倾向 |
|------|----------|----------|
| **任务频率** | 低频（一年几次） | 高频（一天多次） |
| **任务结构化程度** | 强结构化（固定字段、严格校验） | 弱结构化（自由输入、主观判断） |
| **用户认知负担** | 低（一眼能看完） | 高（需要汇总、对比、推理） |
| **错误可逆性** | 低（错了难挽回，如发合同） | 高（错了容易改，如筛选列表） |
| **数据充足性** | 用户数据稀疏、新用户多 | 用户数据充足、冷启动周期短 |
| **用户对 AI 的信任阈值** | 高（涉及钱、客户、合同） | 低（辅助性、内部性） |
| **结果可验证性** | 难（长期才知道对错） | 易（当场能看出来） |

**读矩阵规则：**
- 绝大多数维度倒向"低 AI 倾向" → **本功能不适合 AI 介入，不强行加 AI**
- 绝大多数维度倒向"高 AI 倾向" → **本功能应该 AI 原生，选对应范式**
- 混合 → 选 **范式 D（决策增强）**，因为它对用户控制权伤害最小

## 3. "AI 不介入"的合法理由（4 选 1）

当判定"本功能不适合 AI 介入"时，**必须**从以下四选一，
不接受"暂不考虑"、"以后再说"、"不是重点"：

1. **超出语料方向** — PRD / 原始语料里没有 AI 相关需求，贸然引入会扩展范围。
2. **竞品证据显示此场景 AI 有效性低** — 引用 `/ux-research` 的具体结论（E2 问题三）。
3. **技术前提不具备** — 说明缺少什么（数据量、模型能力、接口、延迟容忍度）。
4. **用户场景不适合** — 结合 B2B 销售的使用处境说明（如：
   在客户面前演示时不能用不稳定的 AI 建议）。

**禁用理由：** "暂时不做"、"后续规划"、"团队还在评估"、
"需要更多数据"——这些都不是设计决策层面的合法理由，
是项目管理层面的拖延。

## 4. AI Native 的四层深度思考（所有 skill 通用）

任何涉及 AI 的设计决策，必须完成四层思考。
**当前工作流的"两层"（产品层 + 交互层）是不足的。**

### Layer A · 产品层：AI 是否重构了决策路径

- 用户当前需要做多少次主动判断？（N）
- 哪些判断可以交给 AI？哪些必须用户保留（业务责任 / 个性化偏好 / 合规要求）？
- 引入 AI 后主动判断压缩到几次？（N')
- 如果 N - N' < 2，AI 在产品层没有实质重构，**重新思考是否应该做**。

### Layer B · 交互层：AI 原生路径长什么样

- 用自然语言 / 预测 / 代理 / 增强四范式中的哪个范式？
- 从"用户触发"到"任务完成"一共几步？
- AI 失败时的 fallback 是什么？用户能否平滑退回到传统路径？
- 如果 AI 原生路径 ≥ 传统路径，**这不是 AI Native，是 AI 累赘**。

### Layer C · 信任层：用户凭什么相信 AI 的输出（新增）

- AI 的输出是否附带**可追溯的依据**（sources、reasoning、引用的数据）？
- 不确定性如何被表达？（百分比 / 高中低 / hedging 语言 / 无）
- 如果 AI 错了，用户多久能发现？发现后多容易纠正？
- 首次看到这个 AI 功能的用户，需要多少次成功经验才会开始依赖它？

**标杆参考：** Perplexity 的 inline citations、Granola 的转录置信度、Claude 的 "I'm
not sure but..." hedging。

### Layer D · 代理层：用户对 AI 动作的控制权（新增，场景 D 必须执行）

- AI 的动作**过程是否可见**？（不是黑箱执行）
- 用户**是否可以暂停**正在执行的 agent？
- 用户**是否可以接管**（从 agent 手里接回控制权）？
- 错误动作**是否可 undo / rollback**？
- Agent 的**授权边界**在哪里？（哪些动作可以自主做，哪些必须用户确认）

**标杆参考：** Cursor Agent 的 plan preview + step approval、Linear 的 AI "review before
apply"、Claude Code 的 `--dangerously-skip-permissions` 反例（默认必须审批）。

## 5. AI 输出的不确定性表达规范

任何展示 AI 结果的 UI，必须选择以下至少一种方式表达不确定性：

| 表达方式 | 适用场景 | 反例 |
|---------|--------|------|
| **显式置信度** 百分比或高/中/低 | 分类、评分、风险预警 | 无置信度直接给结论 |
| **引用依据** 可点击跳转到原始数据 | 摘要、洞察、建议 | 给结论不给来源 |
| **语言 hedging** "可能"、"建议考虑"、"根据..." | 对话式回答 | 用肯定句表达推测 |
| **多候选并列** 给 2-3 个选项让用户挑 | 自动填充、预测建议 | 只给一个结果强制接受 |
| **预览-执行分离** 先展示 AI 打算做什么，用户确认后执行 | agent 动作 | 直接执行后通知 |

**禁止：** 把 AI 输出伪装成"事实"（plain text，无 hedging，无来源，
无置信度）。这是 AI 产品失去用户信任最快的方式。

## 7. 判定格式（所有 skill 产出的标准节）

当 skill 需要在产出文档里声明 AI Native 决策时，必须用以下格式：

```
【AI Native 判定】
范式：[对话式替代 / 嵌入式预测 / 代理式执行 / 决策增强 / 不介入]
判定依据（矩阵）：[填 7 维判定的倾向，如"6高1低"]
四层思考：
  Layer A（产品层）：决策路径 N → N'，压缩 [是/否]
  Layer B（交互层）：路径 N → M 步，比传统路径 [短/持平/长]
  Layer C（信任层）：不确定性表达方式 = [引用依据 / 置信度 / hedging / 多候选 / 预览-执行 / 无]
  Layer D（代理层）：[仅场景D/agent功能填] 过程可见/可暂停/可接管/可回滚 = [是/否]×4
若"不介入"，理由：[超出语料 / 竞品证据 / 技术前提 / 场景不适合]——[一句话说明]
```

## 8. 反模式清单（AI Slop in Interaction）

以下交互模式是 AI 产品的典型 slop，在任何 skill 的产出中一经发现，立即标注并移除：

| 反模式 | 说明 | 正确做法 |
|-------|------|--------|
| **浮球 AI** | 永远悬浮在右下角的"AI 助手"圆形按钮 | AI 嵌入到具体任务上下文 |
| **新标签页 AI** | 点击进入独立的 AI 对话页，脱离任务 | 在任务原地展开 AI |
| **无来源结论** | AI 给出的摘要/洞察/建议没有任何引用依据 | 附带可点击的 sources |
| **无不确定性** | AI 输出用肯定句表达推测，无置信度无 hedging | 选第 5 节的一种方式表达 |
| **单候选强推** | AI 只给一个建议，用户只能接受或完全拒绝 | 给 2-3 个候选或"按我的来" |
| **静默执行** | Agent 直接执行多步动作，用户看不到过程 | 预览→确认→执行，或动作过程可见 |
| **不可撤销** | AI 动作没有 undo，错了用户要手动恢复 | 所有 AI 动作必须可 undo |
| **模糊按钮** | "AI 优化"、"智能处理"、"让 AI 帮你"——不知道点了会发生什么 | 按钮文案具体化："生成本周跟进纪要" |
| **AI emoji 装饰** | 用 ✨🤖💫 装饰 AI 功能 | 用具体图标说明功能 |
| **全屏 modal AI** | AI 交互强制铺满屏幕打断任务 | 侧栏、inline、或可折叠 |
