# AI 专有状态清单

> 共享 reference。被 /ux-brainstorm、/design-brief、/html-prototype 引用。
> 传统的"默认/空/加载/错误/成功"不够用于 AI 功能。本文件定义 AI 专有的 7 个状态。

---

## 为什么要加这些状态

AI 产品的失败模式与传统 SaaS 不同。传统 SaaS
的错误是**明确的**（请求失败 / 数据为空 / 网络断开），AI
产品的错误是**模糊的**（不确定、部分完成、可能错了、在思考）。

如果 UI 只覆盖"默认/空/加载/错误/成功"，
用户会遇到一堆没有对应界面的状态——直接表现就是"这个 AI 功能很怪、
不好用、不敢用"。

**所有涉及 AI 的设计决策必须显式处理以下 7 个状态。**
不适用的要写明"N/A — 原因"，不允许静默跳过。

---

## 状态 1 · 思考中态（Thinking / Streaming）

**区别于传统 loading：** loading 是"服务器在处理，请等待"，thinking 是"AI
正在生成，内容在逐步出现"。两者的 UI 表达完全不同。

**必须解决的设计问题：**
- 用户**现在**能看到什么？（已经生成的部分？打字机效果？骨架屏？）
- 用户**能不能中断**？（对话式 AI 必须能 stop，长时 agent 必须能 pause）
- 等待时长**可预期吗**？（进度条？"预计 X 秒"？还是不确定时长？）

**标准模式：**
- 对话式 AI → 流式输出 + stop 按钮
- 嵌入式预测 → ghost text（灰色预览，按 Tab 接受）
- 代理式执行 → 计划预览 + 分步展开 + pause 按钮
- 决策增强 → 占位骨架屏 + 预计耗时提示

**反例：** 转圈 loading 几秒后突然弹出完整 AI 结果，用户没有过程感，
不知道系统在做什么。

---

## 状态 2 · 低置信态（Low Confidence）

**定义：** AI 有结果，但置信度低于可靠阈值（业务场景决定阈值，B2B 销售通常 85%）。

**必须解决的设计问题：**
- 低置信结果**要不要展示**？（展示但标注 / 不展示改为拒答 / 降级为"参考建议"）
- 置信度**怎么表达**？（百分比 / 高中低 / hedging 语言 / 视觉权重降低）
- 用户**如何提供更多上下文**来提高置信度？

**标准模式：**
- 结果旁显示 `置信度：中` 或 `⚠️ 基于有限数据`
- 视觉权重降低（颜色变灰 / 字号不变但不加粗）
- 附加"提供更多信息"的入口（让用户补充上下文）

**反例：** 对一个只有 1 次联系的客户生成"成单概率 73%"，不标注"数据不足"。

---

## 状态 3 · 拒答态（Decline / Refuse）

**定义：** AI 识别出请求超出自己的能力边界，主动拒绝生成，
而不是硬生成一个低质量结果。

**必须解决的设计问题：**
- 拒答**怎么呈现**？（明确告知 / 建议替代方案 / 不留空白）
- 拒答时**用户下一步怎么办**？（换个问法 / 手动处理 / 联系客服）
- 拒答**会不会让用户觉得产品没用**？（需要配合 onboarding 或说明）

**标准模式：**
- "我对这个问题的数据不足以给出可靠建议，你可以：[选项 A] [选项 B]"
- 避免冷冰冰的 "Error" 或 "Cannot process"
- 保留用户的输入，方便他们换个问法重试

**反例：** AI
对超出能力的请求硬生成一个听起来合理但错误的结果——这是最严重的
slop，会摧毁用户信任。

**标杆：** Claude 会说"I'm not sure, but here's what I can tell you..."；Perplexity 会说"I
couldn't find a reliable source"。

---

## 状态 4 · 部分完成态（Partial Completion，agent 场景）

**定义：** Agent 执行到一半，某些步骤成功、某些失败，整个任务处于中间态。

**必须解决的设计问题：**
- 用户**如何看到**哪些步骤成功、哪些失败？
- 失败的步骤是**可重试**、**可跳过**、还是**整体回滚**？
- 已成功的步骤**是否保留**，还是跟着失败一起回滚？

**标准模式：**
- 步骤清单视图：✅ / ❌ / ⏸️ 三态并存
- 每个失败步骤旁有 `重试` / `跳过` / `手动完成` 三选项
- "整体回滚"按钮可见但不醒目（默认保留已成功步骤）

**反例：** Agent 跑到一半失败，直接弹一个 "Failed" 提示，
用户不知道已经做了什么、还能不能接着做。

---

## 状态 5 · 待 Steer 态（Awaiting User Input / Multiple Candidates）

**定义：** AI 不替用户决定，而是给出 2-3 个候选，等用户挑。

**必须解决的设计问题：**
- 候选**怎么并列**？（卡片 / 列表 / diff view）
- 候选之间的**差异怎么显化**？（关键不同点高亮）
- 用户**怎么表达偏好**（接受某个 / 都不满意 / 混合）？

**标准模式：**
- 2-3 个候选平铺，不隐藏任何一个
- 每个候选配一句话说明"这个版本的特点"
- "都不满意，重新生成" + "按我的来" 两个 fallback 按钮

**反例：** AI 只给一个结果，用户只能"接受"或"拒绝"，没有中间选项。

**标杆：** Midjourney 的 4 张候选图；Cursor 的多方案补全；Notion AI 的
"regenerate" + "make shorter/longer"。

---

## 状态 6 · 幻觉兜底态（Hallucination Recovery）

**定义：** 用户发现 AI 结果是错的，标记了 / 举报了 / 修正了之后的界面。

**必须解决的设计问题：**
- 用户**怎么标记错误**？（一键反馈 / thumbs down / "这是错的"按钮）
- 标记后 AI 是**立即修正**还是**学习后修正**？
- 同样的错误**下次还会不会出现**？（告知用户我们已记录）

**标准模式：**
- 每个 AI 输出旁有 `👍` / `👎` 或 `标记错误` 入口
- 点击"错误"后可选原因（无依据 / 过时 / 理解错意图 / 其他）
- 反馈后有明确的"已记录"响应，避免用户怀疑反馈是否生效

**反例：** AI 说错了，用户找不到纠错入口；或点了反馈按钮没反应。

---

## 状态 7 · Agent 执行中态（Agent Running，场景 D 必备）

**定义：** Agent 正在长时间运行（分钟级 - 小时级），用户可能切换到其他任务。

**必须解决的设计问题：**
- 用户**离开后怎么回来**？（持久化的 agent 状态 / 通知 / 回到现场）
- Agent 运行中**影响哪些其他功能**？（锁定相关数据？允许并行？）
- 运行**失败或超时**如何通知？（push / 邮件 / in-app）

**标准模式：**
- 后台任务中心：显示所有正在运行的 agent 任务
- 每个任务可展开看当前进度
- 完成/失败时用 in-app 通知 + 可选的 email/push

**反例：** Agent 后台跑着，用户关掉页面就丢状态，
或者没有任何通知机制，回来发现不知道跑完没。

## 产出格式（所有 skill 通用）

每个 AI 相关设计决策的"状态覆盖表"必须覆盖以下**12 个状态**（原有 5 个 +
AI 专有 7 个）：

```
| 状态 | 方案处理方式 | 是否需要单独设计 |
|------|-----------|---------------|
| 默认态 | {描述} | {是/否} |
| 空态 | {描述 / N/A：原因} | {是/否} |
| 加载态 | {描述 / N/A：原因} | {是/否} |
| 错误态 | {描述 / N/A：原因} | {是/否} |
| 成功态 | {描述 / N/A：原因} | {是/否} |
| ─── AI 专有 ─── |
| 思考中态 | {描述 / N/A：原因} | {是/否} |
| 低置信态 | {描述 / N/A：原因} | {是/否} |
| 拒答态 | {描述 / N/A：原因} | {是/否} |
| 部分完成态 | {描述 / N/A：原因（非 agent 场景写 N/A）} | {是/否} |
| 待 Steer 态 | {描述 / N/A：原因} | {是/否} |
| 幻觉兜底态 | {描述 / N/A：原因} | {是/否} |
| Agent 执行中态 | {描述 / N/A：原因（非 agent 场景写 N/A）} | {是/否} |
```

**场景判定：**
- 场景 A / B（无 AI 功能） → 只填前 5 个，AI 专有全填"N/A — 本功能不涉及 AI"
- 场景 A / B（含 AI 功能） → 前 5 个 + 思考中/低置信/拒答/待 Steer/幻觉兜底
  必填，部分完成/Agent 执行中 看是否涉及 agent
- 场景 C（评审改版） → 看 ux-audit 报告的范围
- **场景 D（Agent 化改造） → 12 个全部必填，不允许 N/A**

## 与下游 SKILL 的契约

`/html-prototype` 读取 `/design-brief` 产出的"状态覆盖表"并为每个非 N/A
的状态生成对应的 UI。本文件扩展的 7 个 AI 状态在 html-prototype 场景 A/B
的全量状态产出要求中生效。

`/design-review` 后置验收时对照本清单检查：是否所有非 N/A 的状态都在原型里有对应实现。
